Storage Management — Drive Quota, Cleanup, Archive
frozen odio-2026-017 storage google-drive cleanup archive quota
How to use

In the Band tab, tap Settings to see Drive storage usage. The bar shows how many MB/GB of Drive are used by this band's recordings. After a clip is frozen and the band is happy with it, tap 'Clean up source audio' on the clip detail page to delete the raw recording from Drive — the final rendered file is kept and the clip remains fully playable. Tap 'Delete clip' (Recorder only) to permanently remove a clip and all its files from Drive.

Known Limitations

  • Drive API quota query may not reflect real-time usage — cache with 5-minute TTL

  • After cleanup, v1 source audio is gone — if the frozen render was also deleted and the band wants to re-edit, they would need to re-record

  • Drive storage is the recorder's personal quota — shared with all their other Drive content

render
Render — Server-Side FFmpeg on Freeze + Public Share
frozen odio-2026-014 ffmpeg server-side freeze render next.js-api public-share
How to use

On a clip's detail page, tap 'Freeze' on any version to produce the final audio file. The page shows 'Rendering…' for a few seconds, then displays a lock icon and download button. To share publicly: tap the Share icon on a frozen clip and copy the /share/[token] link. Anyone with the link can listen without signing in. To revoke public access, tap Share again and toggle it off.

Known Limitations

  • Vercel /tmp is ephemeral (512MB limit) — source download (~43MB) + FFmpeg output (~10MB) is within limits for typical clips

  • Render requires recorder's valid Drive token from Postgres — if token refresh fails (e.g., recorder revoked access), render fails with a clear error and re-auth prompt

  • @ffmpeg-installer/ffmpeg bundles ~38MB compressed — close to Vercel's function size limit; this is the highest-risk technical component and needs a spike before building

  • Any band Editor role can trigger freeze, but the render always uses the recorder's Drive token — recorder does not need to be signed in at the moment of freeze

Versioning — Version Chain, Waveform Editor, Waveform Player, Freeze
active odio-2026-012 versioning version-history freeze wavesurfer waveform-editor waveform-player localstorage skip-cuts split
How to use

From a clip's detail page, select a version in the Versions section to hear that version's cuts applied. Tapping stamp chips (🔥 ⭐ etc.) in the player seeks to that moment. Tap 'Edit cuts' to open the full-screen editor. In the editor: drag on the waveform to create a red cut region; tap Trim to pre-seed start/end cuts; tap Clear all to reset; tap Split to divide the song at a chosen time position — mark the split point on the waveform, confirm, and Odio creates Part A and Part B as two fully independent new songs, each with their own trimmed audio file in Drive. The original song stays in the session unchanged. After confirming Part A and Part B are good, go to Settings → Manage files / sync to delete the original's source from Drive. Tap 'Start from' pills to load a previous version's cuts as a starting point. Tap 'Submit edit' to save as a new version — description is optional.

Known Limitations

  • Split takes a few seconds on the server (FFmpeg download + trim + upload) — the UI shows a loading state on the confirm button but gives no progress indicator

  • After splitting, the original Song X remains in the session — the band must manually delete it from the Drive page once they've confirmed A and B are good

  • If two members edit the same version simultaneously they both produce the next version number — band sees two versions at the same level and must choose

  • Submit requires internet connection — cuts safe in localStorage but cannot be written to Postgres offline

FFmpeg POC — Validate Bundle Size and Render Pipeline on Vercel
frozen odio-2026-002 ffmpeg poc vercel bundle-size spike render
How to use

This is an internal validation endpoint, not a user-facing feature. To test: GET /api/render/poc?url=<public-audio-url>. The endpoint downloads the audio, applies a test cut (removes seconds 0-10), and returns the trimmed file. Used during development to confirm Vercel can run FFmpeg before building the real render pipeline.

Known Limitations

  • Vercel Hobby function timeout is 10s — this may be too tight for long clips; Pro plan raises to 300s

  • If bundle size fails on Hobby, Vercel Pro ($20/month) is the simplest fix — breaks the $0/month cost target

Multi-Recorder — Any Editor Can Record Into the Band's Drive
frozen odio-2026-008 recording multi-recorder roles permissions google-drive upload
How to use

Any Editor-role band member can record — not just the band creator. Editors see the Record tab in the nav and can start a recording from their own device. The audio automatically lands in the band's Drive folder (owned by the band creator). If the creator's Drive connection expires, a banner appears: 'Drive connection needs renewal — ask [creator name] to re-authorize.' The clip card shows 'Recorded by X' when the recorder is not the band creator.

Known Limitations

  • If the band creator revokes Google Drive access, ALL recording attempts by any Editor fail until the creator re-authorizes — one token serves the whole band; there is no per-member fallback

  • Two Editors recording simultaneously produce two separate clip rows with overlapping timestamps — no merging or conflict detection; both appear in the session normally

  • The creator's refresh_token is stored server-side in the NextAuth accounts table — this security surface already exists for the audio proxy feature; multi-recorder adds no new exposure

Auth — Google OAuth via NextAuth.js
frozen odio-2026-003 auth google-oauth nextauth session drive.file
How to use

Users visit the app and click 'Sign in with Google'. They are prompted to grant Google Drive access (files created by Odio only). After sign-in, they are redirected to the bands home page. Sessions persist across browser restarts. Sign-out via the Band settings page. The /share/[token] public page requires no sign-in.

Known Limitations

  • drive.file scope only covers files the app created under that specific user's account — bandmates stream audio through the server-side proxy (which uses the recorder's stored token), so bandmates do not need Drive scope just to listen

  • Google OAuth tokens expire after 1 hour — NextAuth DB adapter stores refresh_token in Postgres and rotates on expiry; JWT-only strategy is insufficient and must not be used

  • If the recorder revokes Odio's Drive access (in Google account settings), all audio becomes inaccessible to the entire band until re-authorized

Band Space — Band Creation, Invites, Drive Folder, Members
active odio-2026-005 band invite members google-drive postgres qr-code role-management
How to use

Create a band from the home screen. Share the invite link or QR code with bandmates — they sign in with Google and join as Editors. After someone joins, their email appears in Settings → Members. Share the Drive folder with that email from Google Drive (one-time step). In Settings, use the pill buttons to switch a member between Editor and Member role. Tap Remove to remove a member (confirmation required). Non-Recorder members can Leave band from their own member row (confirmation required). Editors and Recorders can open 'Manage files / sync' from the Storage section to see all Drive files grouped by session. Hit 'Sync' to check which files still exist in Drive (green dot = ok, red dot = missing). Delete a source file to free Drive space while keeping the clip and its versions. Delete a clip to remove it entirely from Odio and Drive.

Known Limitations

  • New members cannot hear audio until the recorder manually shares the Drive folder with their Google email

  • If the recorder loses access to their Google account, band audio becomes inaccessible — no recovery path without exporting files first

  • Drive sync checks file existence but does not verify file integrity or playback quality

  • Deleting a clip from the Drive page removes it globally — all band members lose access immediately

Clip Detail — Coordinator, Actions Panel, Freeze, Share, Stage, Cleanup
active odio-2026-013 clip-detail coordinator freeze share stage cleanup actions version-selection
How to use

Open a clip from the session list. The waveform player at the top always plays the selected version. The Edit tab is your main workspace: tap 'Edit cuts' at the top to open the full-screen editor; tap any version card to switch what you hear (latest version is listed first, original at the bottom). Once you're happy with a version, tap Freeze to lock it — you'll be prompted to generate a share link right away. The Team tab is for post-edit collaboration: it shows who voted for what on the current version and the full comment thread. Tap 'change' in the voting header to switch which version the band is voting on.

Known Limitations

  • Share link generation requires clip to be frozen — members cannot share a clip-in-progress

  • publicToken is stored in Postgres; if the tokens table is cleared, existing share links break

  • Unfreeze does not delete the rendered Drive file — it simply sets frozen=false. The rendered file remains in Drive until cleanup or clip delete.

Collaboration — Votes, Comments, Stamp Heat Map, Activity
frozen odio-2026-016 votes comments stamps heat-map activity polling postgres
How to use

On a clip's detail page, the Versions tab shows all edits; the Comments tab shows the discussion. Vote on any version by tapping Keep, Revise, or Pass — the tally updates for all band members within seconds. Leave a comment in the text field (optionally tap the waveform first to attach a timestamp). When everyone has voted Keep on the same version, a banner appears prompting the Recorder or an Editor to freeze it. Comments can be edited or deleted by their author (long-press or tap the '...' menu).

Known Limitations

  • No push notifications — members need to open the app to see new votes/comments

  • Polling at 5-10s means activity is not instant — acceptable for post-jam async workflow

  • Stamp heat map is derived from recording-time stamps only (not from member actions during review)

Upload Banner — Persistent Upload Progress Across All Pages
active odio-2026-010 upload banner progress resilience indexeddb retry fixed-ui
How to use

After stopping a recording, a banner appears at the top of every screen showing upload progress. If the upload completes successfully, the banner disappears automatically. If the connection drops, the banner shows 'Upload paused' with a Retry button — tap it once reconnected. If the band creator's Drive connection expires, the banner shows an error with a 'Save file' button as a last resort to download the raw recording to your device.

Known Limitations

  • Banner occupies top space — sticky headers below must consume --upload-banner-h CSS variable; new pages with their own sticky headers need to adopt this pattern

  • On desktop the banner is offset md:left-[220px] to avoid the sidebar; if the sidebar width changes, this value must be updated manually

  • Progress bar accuracy depends on IndexedDB byteOffset being updated frequently during upload; if updates are sparse, the bar may jump

  • 'Save file' downloads the raw browser-recorded blob (WebM on Chrome, AAC/MP4 on Safari) — not the transcoded AAC that would normally be stored in Drive

Recording Resilience — IndexedDB Safety Net, Upload Retry, Save-to-Device Fallback
frozen odio-2026-009 recording indexeddb resilience offline retry upload fallback
How to use

Recording resilience is automatic and invisible when things work. If an upload fails (network drops, Drive token expires), a banner appears at the top of the screen: 'You have [N] recording(s) waiting to upload'. Tap 'Retry' to resume when reconnected. Tap 'Save to device' to download the raw audio file to your phone or computer if Drive is permanently unavailable. Tap 'Discard' (with confirmation) to remove a pending recording from recovery. When the app reopens after a crash, pending recordings are detected automatically.

Known Limitations

  • IndexedDB is cleared when the user clears browser storage ('Clear site data') — the blob would be lost; this is unavoidable without server-side buffering and is an edge case the user controls

  • Background Sync (automatic retry when tab is closed and connection restored) is not available on iOS Safari — recovery on iOS requires opening the app manually

  • Drive resumable upload session URLs expire after 7 days — a blob stranded for > 7 days will need a fresh upload session, not a resume; the implementation must detect this (404 on session URL) and fall back to a new session

  • Storage pressure on low-end devices: a 43MB blob may trigger the browser's storage eviction policy on devices with < 1GB free — 'Save to device' should be surfaced proactively for large recordings

  • PostRecordScreen.doUpload() is a standalone async function that writes to IndexedDB directly without going through the Zustand store's action methods — fixed by calling usePendingUploadsStore.getState().refresh() on every status-changing path so the in-memory store stays in sync with IndexedDB and the banner dismisses immediately
