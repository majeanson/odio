odio-2026-017 Frozen storage P3

Storage Management — Drive Quota, Cleanup, Archive

Problem

Audio files in Google Drive grow over time (~43MB per raw clip, ~10MB per frozen render). The recorder's free Drive tier (15GB) supports roughly 350 raw clips before action is needed. The app must surface this, and provide a clean way to free space after songs are finalized.

Analysis

Storage indicator shown in band settings and session list: total Drive storage used by Odio files for this band (queried via Drive API quotas endpoint). Post-freeze cleanup: once a clip is frozen and the band is satisfied, the source .aac (raw recording) can be deleted from Drive — the frozen render and Postgres metadata remain. Archive: delete all versions of a clip from Postgres + Drive (both source and render). Bulk cleanup: session-level 'Archive all frozen clips' removes source audio for all frozen clips in a session at once.

Decisions
  1. 1. Cleanup is explicit and manual — no automatic deletion 2026-04-10
    Automatic deletion after freeze could destroy recordings the band wanted to keep. Storage cleanup is a deliberate action. The app nudges (shows quota, shows how much can be freed) but never deletes automatically.
Implementation
  1. app/api/bands/[bandId]/storage/route.ts: GET — fetches Drive API about endpoint using creator's token (returns storageQuota.usage). Also queries Postgres for all driveFileId + finalDriveFileId for this band's clips. Returns { usedBytes, limitBytes, fileCount }. Cached in memory for 5 minutes.
  2. app/api/clips/[clipId]/cleanup/route.ts: POST — deletes clip.driveFileId from Drive (source .aac). Sets clip.driveFileId=null. Requires clip.frozen=true.
  3. app/api/clips/[clipId]/route.ts DELETE: RECORDER only. Deletes clip from Postgres (cascades to versions, votes, comments). Deletes both driveFileId and finalDriveFileId from Drive via lib/drive.ts deleteFile.
  4. components/bands/BandSettingsClient.tsx: fetches /api/bands/[bandId]/storage on mount. Renders storage bar (used / total). Lists members with role management. formatBytes helper: converts bytes to human-readable MB/GB string.
  5. components/clips/ClipActionsClient.tsx: shows 'Clean up source audio' button when clip.frozen=true and clip.driveFileId is not null. Shows 'Delete clip' button (RECORDER only). Confirmation bottom sheets for both.
Known Limitations
  • Drive API quota query may not reflect real-time usage — cache with 5-minute TTL
  • After cleanup, v1 source audio is gone — if the frozen render was also deleted and the band wants to re-edit, they would need to re-record
  • Drive storage is the recorder's personal quota — shared with all their other Drive content
Success Criteria
  • Band settings shows Drive storage used by Odio files (MB / GB)
  • Post-freeze: 'Clean up source audio' button removes raw .aac from Drive, keeps frozen render
  • Confirmation dialog shows exactly what will be deleted and what will remain
  • Archive clip: removes Postgres records + Drive files; cannot be undone
  • Bulk cleanup: 'Clean up all frozen clips in session' removes source audio for all frozen clips at once
  • Storage indicator updates after cleanup
  • Cleaned-up clips remain playable via their frozen render file

odio-2026-014 Frozen render P2

Render — Server-Side FFmpeg on Freeze + Public Share

Problem

When a version is frozen, a real audio file must be produced — so the band can download it, share it outside the band, or export it. This cannot happen client-side. The frozen clip should also be shareable publicly without requiring sign-in.

Analysis

Next.js API route POST /api/render: fetch source .aac from Drive (via recorder's stored token), run FFmpeg atrim+concat to apply cutMarks, output final .aac at 192kbps, upload result as {clipId}-final.aac to Drive. Update clips row: frozen=true, frozenVersionId, frozenDriveFileId. Progress communicated via Postgres poll — client polls GET /api/clips/{id} every 3s until frozen=true. For public share: frozen clips get a public Drive permission set ('anyone with link can read') and a short public token stored in Postgres. Public page /share/[token] fetches clip metadata from Postgres and streams audio from the Drive public URL — no sign-in required.

Decisions
  1. 1. Server-side FFmpeg only — no client-side WASM 2026-04-10
    WASM FFmpeg is slow on mobile (~3x slower than native), memory-constrained, and adds bundle size. Server-side runs once per clip lifetime in < 10s on Vercel. Acceptable trade-off.
  2. 2. Public share via Drive public permission + Postgres token 2026-04-10
    A frozen clip can be shared outside the band (with a producer, a friend). Setting Drive file permission to 'anyone with link' gives a direct audio URL. Odio stores a short token in Postgres mapping to the clip, so the public URL is /share/[token] (not a raw Drive ID), giving control over revocation.
Implementation
  1. app/api/clips/[clipId]/freeze/route.ts: POST — Editor+ role. Validates versionId. Fetches creator's Drive token from NextAuth accounts table. Downloads source .aac to /tmp. Runs fluent-ffmpeg atrim+aconcat with cutMarks from the version. Uploads output as {clipId}-final.aac to band's Drive folder. Sets clip.frozen=true, clip.frozenVersionId, clip.finalDriveFileId. Returns updated clip.
  2. app/api/clips/[clipId]/unfreeze/route.ts: POST — RECORDER only. Sets clip.frozen=false, clip.frozenVersionId=null. Keeps finalDriveFileId.
  3. app/api/clips/[clipId]/route.ts: PATCH handles generatePublicToken (requires frozen=true — sets publicToken=crypto.randomUUID(), sets Drive public permission) and removePublicToken (clears publicToken, removes Drive public permission).
  4. app/share/[token]/page.tsx: public page — no auth. Fetches clip by publicToken from Postgres. Renders waveform player with Drive public audio URL + clip name + stamps.
  5. FFmpeg atrim+aconcat filter chain: for each non-cut segment, atrim=start=X:end=Y, asetpts=PTS-STARTPTS, then aconcat=n=N. Output: -b:a 192k -f adts (AAC).
  6. lib/drive.ts: downloadToTemp(fileId, token) → /tmp/{fileId}.aac. uploadFromTemp(path, folderId, token) → new driveFileId. setPublicPermission(fileId, token). removePublicPermission(fileId, permissionId, token).
Known Limitations
  • Vercel /tmp is ephemeral (512MB limit) — source download (~43MB) + FFmpeg output (~10MB) is within limits for typical clips
  • Render requires recorder's valid Drive token from Postgres — if token refresh fails (e.g., recorder revoked access), render fails with a clear error and re-auth prompt
  • @ffmpeg-installer/ffmpeg bundles ~38MB compressed — close to Vercel's function size limit; this is the highest-risk technical component and needs a spike before building
  • Any band Editor role can trigger freeze, but the render always uses the recorder's Drive token — recorder does not need to be signed in at the moment of freeze
Success Criteria
  • Freeze triggers render within 2s of user confirmation
  • Rendered file available within 10s for a typical 2-8 min output clip
  • Clip page shows lock icon and download button after successful render
  • Render failure shows clear error with retry option
  • Public share toggle: generates /share/[token] URL, shareable outside the band
  • Public page: waveform player + stamps + clip name + date — no other session data exposed
  • Public listener does not need a Google account
  • Download of rendered file available on clip page (authenticated) and public page

odio-2026-012 Active versioning P2

Versioning — Version Chain, Waveform Editor, Waveform Player, Freeze

Problem

A raw recording needs to evolve from v1 (messy, includes setup and dead air) to vFinal (clean, trimmed) through collaborative edits — with full history and a clear end state (freeze). The band needs to listen to any version, compare versions, and hear exactly what a cut sounds like before committing.

Analysis

Two dedicated components with separate concerns:

WaveformPlayer (clip detail page) — pure playback. Receives activeCuts from parent (ClipDetailClient). Visualises cuts as blue non-draggable regions (pointerEvents: none on regions so they don't block click-to-seek). Skip-cuts during playback via timeupdate handler. H:MM:SS clock + virtual duration display. Stamp chips are tappable — each seeks to that timestamp. Big 64px play button. finish event resets cursor to start (ws.setTime(0)) so the amber cursor doesn't pin at the right edge after playback ends.

WaveformEditor (edit page, always edit mode) — cut creation only. Custom pointer overlay (absolute inset-0, touch-none, z-20 inside WaveformCanvas) handles tap-to-seek and drag-to-create. Cut bands are pure CSS divs (no RegionsPlugin). useCutInteraction manages the drag state machine; edge drag resizes existing cuts. Trim shortcut (pre-seeds start+end cuts). Split mode (mark a time position, confirm → creates TWO new clips: Part A (0→splitMs) and Part B (splitMs→end); original clip left completely unchanged as a reference; navigates to session page showing all three clips). Submit → new version in Postgres, router.back().

Non-destructive split with immediate Drive files: original clip is never mutated. Splitting Song X into Part A and Part B: the split API downloads X's source once, runs FFmpeg twice in parallel to produce two trimmed files (A: 0→splitMs, B: splitMs→end), uploads both to Drive with their own names, and creates two new Clip rows with their own driveFileId + correct sourceDurationMs + empty v1 cutMarks (the trimmed file is their raw). The original clip is untouched. After confirming that A and B are good, X's source can be deleted from Drive to free space. No provenance UI is shown on split children (no scissors icon, no 'Split into' card) — they are fully independent songs from the user's perspective. sourceClipId exists in the DB for data integrity but is not surfaced in the UI.

ClipDetailClient — owns selectedVersionId state. Feeds activeCuts to WaveformPlayer. Feeds activeVersionId to CollaborationSection for version-aware voting. Version list as large tappable cards. Clicking active version keeps it selected (no deselect/null state). Edit link to /edit page.

Versions are metadata only (cutMarks JSON) pointing at the same source audio. Source audio never mutated. Linear chain. localStorage safety net for in-progress edits.

Decisions
  1. 1. WaveformPlayer and WaveformEditor are separate components 2026-04-11
    A single component that handles both playback and editing requires mode-switching logic, conditional overlays, and competing event handlers. Separate components have clear single responsibilities: Player = listen, Editor = cut. The edit page is always in edit mode; no toggle needed. The detail page never needs editing. Each component can be reasoned about and tested independently.
  2. 2. interact:false + single overlay div for all pointer handling (no enableDragSelection, no RegionsPlugin) 2026-04-15
    WaveSurfer's enableDragSelection registers a permanent pointer overlay that consumes all pointer events — including click-to-seek. Once called, seeking by clicking is broken. interact:false disables WaveSurfer's built-in event handling entirely. A single overlay div (absolute inset-0, touch-none) inside WaveformCanvas intercepts all pointer events and forwards them to useCutInteraction (trim mode) or useSplitInteraction (split mode).
  3. 3. Split produces independent trimmed Drive files immediately (not virtual clips sharing source) 2026-04-11
    Original model: Part A and Part B shared Song X's Drive file and applied cuts virtually — correct waveform, wrong duration in clip list (both showed full 9-min duration instead of 4 min and 5 min). Trimmed files fix duration display without special-casing. They also make Part A and Part B genuinely independent: they can be frozen separately without needing X's file, and X's source can be safely deleted from Drive once A and B are confirmed. FFmpeg at split time costs a few seconds of processing but eliminates a class of edge cases.
  4. 4. No provenance UI on split children 2026-04-11
    Showing scissors icons and a 'Split into' provenance card on Song X implied that Song X was consumed or changed by the split. The correct mental model is: X exists unchanged, A and B are new songs. Their name might hint at their origin (e.g. 'Groove - Part A') but they are first-class independent songs from the moment they're created. Provenance exists in the DB (sourceClipId) for integrity but is not surfaced in the UI.
  5. 5. Split navigates to session page, not back to clip detail 2026-04-11
    After a split, three clips exist (original + Part A + Part B). router.back() would return to the original clip's detail — neither Part A nor Part B would be visible. router.push(sessionHref) lands on the session clip list where all three are visible and accessible.
  6. 6. No version deselect — clicking active version keeps it selected 2026-04-11
    Deselecting a version set selectedVersionId to null, which caused the player to show 'no cuts' (raw audio). This 'null = raw audio' state had no UI label and was visually identical to a version with no cuts. Confusing. Clicking the active version now keeps it selected — the player always reflects a known version.
  7. 7. Versions are metadata only — not audio files 2026-04-10
    Audio duplication per version is prohibitively expensive (43MB+ per clip). Versions are cut instructions (cutMarks JSON). Source audio stored once in Drive. Real audio file produced only on freeze.
  8. 8. localStorage for in-progress edit safety 2026-04-10
    Full offline sync adds complexity. The real risk is a browser crash mid-edit. localStorage persists cut marks across crashes. Key: odio:draft:{clipId}. Cleared on successful submit.
Implementation
  1. components/clips/WaveformPlayer.tsx: pure player component. Props: clipId, sourceDurationMs, activeCuts[], stamps[]. Regions rendered with pointerEvents:none (display only, don't intercept seek clicks). timeupdate handler: position tracking (currentTimeMsRef + setCurrentTimeMs) runs unconditionally — isPlaying() guard wraps only the cut-skip section so currentTimeMs updates correctly even while paused (needed for re-seek before play). finish event: ws.setTime(0) resets cursor to start. Stamp chips are <button> elements that call ws.setTime(). No enableDragSelection.
  2. components/clips/WaveformEditor.tsx: edit-only component. Props: clipId, sourceDurationMs, frozenVersionId, initialVersions, stamps, sessionHref. Custom pointer overlay (onPointerDown/Move/Up + setPointerCapture, touch-action:none) handles tap-to-seek + drag-to-create. Trim/Clear all/Split tool buttons. Split confirmation sheet updated to reflect Drive file creation ('Each part gets its own audio file in Drive. Takes a few seconds — hang tight.'). Split navigates to sessionHref after success. Auto-saves draft on every cut change. router.back() after version submit. No provenance UI (splitInto prop removed, scissors icon removed from 'Start from' version buttons).
  3. components/clips/ClipDetailClient.tsx: coordinator. Owns selectedVersionId state (auto-selects latest, no deselect). Passes activeCuts to WaveformPlayer. Passes activeVersionId to CollaborationSection. Actions section hidden when !canEdit && !frozen.
  4. app/(auth)/.../clips/[clipId]/page.tsx: server component — loads clip, versions, stamps, votes, comments, membership. Renders ClipDetailClient.
  5. app/(auth)/.../clips/[clipId]/edit/page.tsx: server component — loads clip, versions, stamps, splitInto (select id, name). Auth guard (MEMBER redirect + frozen redirect). Maps splitInto and passes to WaveformEditor.
  6. app/api/clips/[clipId]/versions/route.ts: POST creates a new version.
  7. app/api/clips/[clipId]/split/route.ts: POST { splitMs } — downloads X's source audio once, runs renderAudioWithCuts twice in parallel (Part A: atrim tail, Part B: atrim head), uploads both trimmed files to Drive with distinct names, creates two Clip rows with own driveFileId + correct sourceDurationMs + empty v1 cutMarks. Both have sourceClipId pointing to the original. Original clip is completely unchanged. Returns { clipA: {id, name}, clipB: {id, name} }. Vercel runtime: nodejs, maxDuration: 300.
  8. lib/utils.ts: formatPosition(ms) → H:MM:SS always. formatDuration(ms) → M:SS. calcResultDuration(sourceDurationMs, cuts) → virtual duration after removing cut segments.
  9. prisma/schema.prisma: Clip.sourceClipId String? (nullable) + SplitFrom self-relation (sourceClip / splitInto) with onDelete: SetNull — deleting a child clip does not cascade to the original.
Known Limitations
  • Split takes a few seconds on the server (FFmpeg download + trim + upload) — the UI shows a loading state on the confirm button but gives no progress indicator
  • After splitting, the original Song X remains in the session — the band must manually delete it from the Drive page once they've confirmed A and B are good
  • If two members edit the same version simultaneously they both produce the next version number — band sees two versions at the same level and must choose
  • Submit requires internet connection — cuts safe in localStorage but cannot be written to Postgres offline
Success Criteria
  • Version list shows full chain: v number, description, result duration, cut count
  • Selecting a version in the detail page updates the waveform player to skip that version's cuts
  • Click-to-seek works on the waveform — clicking anywhere moves the playhead, play starts from that position
  • Stamp chips in the player are tappable and seek to that timestamp
  • After playback ends, cursor resets to start (no amber pin at right edge)
  • Submit button is always active — description is optional
  • Any member can create a version from any existing version
  • Trim shortcut pre-seeds start+end cut marks
  • Split: marks a time position → FFmpeg trims two independent Drive files (Part A: 0→split, Part B: split→end) → creates two new clips each with their own driveFileId and correct sourceDurationMs → original clip unchanged → navigates to session page showing all three clips
  • Split children show correct durations (4 min / 5 min) in the clip list — not the full original duration
  • Split children are fully independent songs with no visible lineage to the original (no scissors icon, no provenance card)
  • Drag on the edit waveform creates a cut region
  • Freeze locks the version and triggers render (see feat-render)
  • In-progress edits survive browser crash via localStorage draft
  • Draft banner shown on editor reopen with Resume / Dismiss

odio-2026-002 Frozen render P1

FFmpeg POC — Validate Bundle Size and Render Pipeline on Vercel

Problem

@ffmpeg-installer/ffmpeg bundles the FFmpeg binary at ~38MB compressed. Vercel's serverless function size limit is ~50MB (250MB uncompressed). This is the single highest-risk technical constraint in the project. If the bundle is too large or the render takes too long (Vercel function timeout: 10s on Hobby), the entire render architecture needs to change before feat-render can be built. This POC must validate or invalidate the approach before any production code is written.

Analysis

The POC validates three things in isolation:

  1. Bundle size: Does a Next.js API route that imports @ffmpeg-installer/ffmpeg + fluent-ffmpeg deploy successfully to Vercel Hobby without hitting the 50MB compressed / 250MB uncompressed limit? The binary alone is ~38MB compressed — leaving ~12MB for the rest of the function bundle.
  1. Cold start: Vercel serverless functions have a cold start on first invocation. A 38MB binary means a large cold start. Is the first render under 30 seconds? Subsequent renders (warm function) should be under 10s for a typical 2–8 min clip output.
  1. Render correctness: Does a fluent-ffmpeg atrim+concat command correctly produce a trimmed AAC file from a source AAC file on Vercel's runtime environment?

If bundle size fails: alternatives are (a) Vercel Pro (1GB function size limit, $20/month — viable), (b) a dedicated render worker (Railway, Fly.io free tier), (c) client-side WASM FFmpeg (slow on mobile, acceptable for desktop-only), or (d) a cloud transcoding service (AWS MediaConvert — adds cost and complexity).

If timeout fails: the 10s Vercel Hobby limit may be too tight for longer clips. Vercel Pro raises the limit to 300s. The POC should test with a worst-case 30-min source clip.

The POC is a standalone Next.js route — not integrated into the app — that accepts a test audio file and returns the trimmed output.

Decisions
  1. 1. POC before any production render code 2026-04-10
    @ffmpeg-installer/ffmpeg is the highest-risk dependency in the entire project. If it fails on Vercel Hobby, the render feature needs a completely different architecture. Building feat-render on top of an unvalidated assumption would waste significant implementation work. The POC is a 2-4 hour investment that prevents a potential full architectural rethink later.
Implementation
  1. Created GET /api/render/poc route that accepts ?url= param pointing to a source audio file.
  2. Installed @ffmpeg-installer/ffmpeg + fluent-ffmpeg as production dependencies.
  3. Route downloads source audio to /tmp, runs fluent-ffmpeg atrim+aconcat filter with a hard-coded test cut (0-10s removed), outputs trimmed .aac.
  4. Returns rendered file as application/octet-stream response.
  5. Result: bundle deploys on Vercel Hobby within size limit. Cold start ~8s, warm render ~3s for a 5-min clip. Architecture validated.

ComponentFile: app/api/render/poc/route.ts

Known Limitations
  • Vercel Hobby function timeout is 10s — this may be too tight for long clips; Pro plan raises to 300s
  • If bundle size fails on Hobby, Vercel Pro ($20/month) is the simplest fix — breaks the $0/month cost target
Success Criteria
  • Next.js project with @ffmpeg-installer/ffmpeg + fluent-ffmpeg deploys to Vercel Hobby without bundle size error
  • Deployed function executes an atrim+concat operation on a test .aac file
  • Output file is a valid .aac file playable in browser
  • Cold start + render time for a 2-8 min clip is under 30s total (10s preferred)
  • Render time for a 30-min source clip with multiple cut regions is measured and documented
  • If any constraint fails: document exactly which limit was hit and the recommended alternative with cost/complexity trade-off

odio-2026-008 Frozen recording P2

Multi-Recorder — Any Editor Can Record Into the Band's Drive

Problem

On jam night, the person who created the band may not always be the one with a phone free to record. Any Editor-role band member should be able to hit Record and have the audio land in the band's shared Drive — without requiring each member to set up their own Drive folder or the band creator to be physically present.

Analysis

The existing recording upload flow already makes multi-recorder architecturally trivial: when a member hits Record, the server's /api/upload/init route generates a Google Drive resumable upload session URL using the band creator's stored refresh_token (saved via NextAuth DB adapter). The member's browser then uploads the audio blob directly to the creator's Drive — the member's own Drive is never involved. The audio lands in the correct band Drive folder regardless of who recorded it.

What needs to change is a role-permission gate: currently /api/upload/init checks that the caller is the band creator (Recorder role). Expanding this to allow Editor-role members is the full scope of the server-side change.

On the client: the Record tab in the bottom nav is shown only to Recorder and Editor roles. Member-only users do not see the Record tab.

One critical edge case: the creator's Drive token can expire. The server must attempt a token refresh before generating the upload session URL. If the refresh fails (e.g., creator revoked app access), the error must be surfaced clearly — 'Band creator needs to re-authorize Drive access' — with a notification mechanism to alert the creator.

The resulting clip row in Postgres records both the band creator (driveOwnerEmail) and the actual recording member (recordedByEmail) so the activity feed can correctly show 'Recorded by [member]'.

Decisions
  1. 1. Reuse creator's stored token — no per-member Drive setup 2026-04-10
    drive.file OAuth scope only grants access to files the app created under that specific user's account. A member's token cannot write to the creator's Drive folder. The creator's refresh_token (already stored by NextAuth DB adapter for the audio proxy feature) solves this: the server uses the creator's token to generate the Drive upload session URL, and the member's browser uploads directly to Drive using that URL. No additional OAuth consent or per-member Drive setup needed.
  2. 2. Editor role can record — Member role cannot 2026-04-10

Recording Resilience — IndexedDB Safety Net, Upload Retry, Save-to-Device Fallback

Problem

A recording is a one-time, irreplaceable event. Currently, if the internet drops during upload or the band creator's Drive token is invalid, the audio blob — which only exists in memory — is permanently lost when the user navigates away or the tab closes. A live jam session cannot be re-played. The recording must survive any upload failure and remain recoverable until it reaches Drive.

Analysis

The audio blob lives in RAM from record-start until upload completes. If anything breaks during the upload window — network drop, Drive token expiry, browser crash, tab close — the blob is gone. This is the highest-risk data loss scenario in the app.

The fix is a two-layer safety net:

Layer 1 — IndexedDB persistence (immediate): The moment recording stops and the blob is assembled, write it to IndexedDB before starting the upload. IndexedDB stores binary blobs up to the device's available storage (typically several GB on modern phones), survives page refresh and browser restarts, and is cleared only explicitly. Key: odio-pending-upload-{tempId}. Also store: tempId, recordedAt, durationMs, bandId, sessionId, blob.

Layer 2 — Resumable upload retry: Google Drive resumable uploads natively support resume from a byte offset. If the upload is interrupted mid-transfer, the server stores the upload session URL in the pending record. On retry, the client queries Drive for the current byte offset and resumes from there — no re-upload of already-transferred bytes.

Decisions
  1. 1. IndexedDB as the primary persistence layer for unuploaded audio blobs 2026-04-10
    localStorage cannot store binary blobs. The File System Access API requires explicit user permission. IndexedDB is the only browser-native store that (a) handles large binary data, (b) persists across sessions without user action, and (c) is available in all browsers including iOS Safari 10.3+. A 43MB blob fits easily — IndexedDB storage limits are typically 50-80% of available disk space.
  2. 2. Resumable upload — save session URL to IndexedDB for byte-accurate retry 2026-04-10
    A naive retry re-uploads the entire blob from byte 0. For a 43MB file on a slow mobile connection, this could fail again at the same spot. Google Drive resumable uploads let you query the committed offset and resume from there. Storing the uploadSessionUrl in IndexedDB alongside the blob enables true resume, not restart.
  3. 3. 'Save to device' as the last-resort fallback — never lose the audio 2026-04-10
    If Drive is permanently unavailable (creator deleted their account, Drive quota exceeded, token irrecoverable), the band should still get their recording. A browser download gives them the raw file. They lose the Odio versioning workflow but not the audio. The guarantee is: if you recorded it, you can get the file.
  4. 4. Check IndexedDB for pending recordings on every app mount 2026-04-10
    Background Sync is not reliable on iOS Safari. The safest recovery path is: when the app is opened, check for stranded blobs and surface them immediately. This catches the crash-during-upload, tab-closed-early, and came-back-later scenarios without relying on service worker scheduling.
Implementation
  1. lib/pendingUploads.ts: IndexedDB helpers using idb library. Database 'odio-recordings', object store 'pending-uploads'. Functions: savePendingUpload(upload), getPendingUploads(), deletePendingUpload(tempId), updatePendingUpload(tempId, patch).
  2. hooks/useRecorder.ts: calls savePendingUpload immediately on MediaRecorder onstop before any network request.
  3. hooks/usePendingUploads.ts: checks IndexedDB on mount, drives retry logic, manages upload state machine (idle/uploading/paused/error), listens to window 'online' event for auto-retry.
  4. components/recording/PendingUploadsBanner.tsx: recovery banner shown when usePendingUploads reports pending items. Shows count, retry button, save-to-device button, discard button with confirmation.
  5. app/(auth)/layout.tsx: renders PendingUploadsBanner at top of all authenticated pages.
  6. lib/uploadAudio.ts: manages the Drive resumable upload from browser — tracks byte offset, handles chunking, updates IndexedDB with uploadSessionUrl on each successful chunk.
  7. 'Save to device': URL.createObjectURL(blob) + programmatic anchor download. Extension .aac or .webm based on mimeType.
Known Limitations
  • IndexedDB is cleared when the user clears browser storage ('Clear site data') — the blob would be lost; this is unavoidable without server-side buffering and is an edge case the user controls
  • Background Sync (automatic retry when tab is closed and connection restored) is not available on iOS Safari — recovery on iOS requires opening the app manually
  • Drive resumable upload session URLs expire after 7 days — a blob stranded for > 7 days will need a fresh upload session, not a resume; the implementation must detect this (404 on session URL) and fall back to a new session
  • Storage pressure on low-end devices: a 43MB blob may trigger the browser's storage eviction policy on devices with < 1GB free — 'Save to device' should be surfaced proactively for large recordings
  • PostRecordScreen.doUpload() is a standalone async function that writes to IndexedDB directly without going through the Zustand store's action methods — fixed by calling usePendingUploadsStore.getState().refresh() on every status-changing path so the in-memory store stays in sync with IndexedDB and the banner dismisses immediately
Success Criteria
  • Audio blob written to IndexedDB immediately on recording stop, before upload begins
  • IndexedDB entry keyed by tempId; stores blob + recordedAt + durationMs + bandId + sessionId + uploadSessionUrl (if partially started)
  • If upload succeeds: IndexedDB entry cleared; no other change
  • If network drops during upload: banner shows 'Upload paused — waiting for connection'; auto-retries on window 'online' event; resumable upload continues from last committed byte
  • If creator's Drive token is invalid: banner shows 'Drive connection needs renewal — ask [creator name] to re-authorize'; IndexedDB blob retained; retry button enabled once token is refreshed
  • If recording member's own session token expires mid-upload: client silently calls getSession() to refresh; if refresh succeeds, upload resumes automatically; if refresh fails, banner shows 'Session expired — sign in again to resume'; after re-sign-in, IndexedDB blob is still present and recovery banner activates
  • On app reopen: check IndexedDB for pending recordings on mount; if found, show recovery banner: 'You have [N] recording(s) waiting to upload'
  • Recovery banner persists until all pending recordings are uploaded or manually discarded
  • 'Save to device' fallback button available on any failed upload: triggers browser download of blob (.webm or .aac depending on browser)
  • User can discard a pending recording from the recovery UI (with confirmation: 'Discard this recording? It cannot be recovered.')
  • 'Pending uploads' count visible in the Record tab (badge) and on the session screen
  • Successful retry creates the clip in Postgres and Drive exactly as a normal upload would

Odio — Collaborative Jam Session Recorder

Problem

Musicians lose the best moments from their jam sessions. When they do record, the raw file is long, unedited, buried — nobody goes back and cleans it up. No tool lets a small band record on any device, pass the recording around in the car on the way home, each person trimming a song into its final form, and arrive with a clean versioned archive — all from a browser, no install, no DAW.

Analysis

Pure web PWA (Next.js). No native app. Browser MediaRecorder API covers recording on iOS Safari 14.5+, Android Chrome, and desktop. The core primitive is versioning: every clip starts as v1 (raw recording), band members create new versions by marking cut regions, and any version can be frozen — triggering a server-side FFmpeg render. Versions are metadata only (cutMarks + annotations JSON) pointing at the same source audio. No simultaneous editing; linear versioning. Storage split: audio files (.aac) in Google Drive (recorder's account, free), all structured metadata (sessions, clips, versions, votes, comments, stamps) in a free-tier Postgres (Neon or Vercel Postgres). No Supabase. Auth: NextAuth.js + Google OAuth (drive.file scope). Real-time: polling every 5-10s. Edit state in localStorage as safety net; submit requires connection. External services: Google + free Postgres. Cost: $0/month.

Decisions
  1. 1. Web-only PWA — no native app 2026-04-10
    Browser MediaRecorder API works on iOS Safari 14.5+, Android Chrome, and desktop. No install removes friction for bandmates who only need to edit/review. The 'in the car editing' use case requires web access — native would require app install on every bandmate's phone.
  2. 2. Google Drive for audio, Neon Postgres for metadata — no Supabase 2026-04-10
    Supabase Storage costs accumulate at ~100MB/session. Google Drive is free (15GB) for audio. Structured metadata (sessions, clips, versions, votes, comments) belongs in a relational DB for querying, not in JSON sidecars with ETag concurrency management. Neon Postgres free tier (512MB) handles years of band metadata. Both services are $0.
  3. 3. NextAuth DB adapter required — not JWT-only 2026-04-10
    The audio proxy route serves the recorder's Drive files to all band members. To do this, the server needs the recorder's valid refresh token when any member makes a request — not just when the recorder is logged in. NextAuth's DB adapter stores access_token and refresh_token in Postgres per user, enabling server-side token lookup by user email. JWT-only strategy cannot support this cross-user token access.
  4. 4. Versioning as core primitive — versions are metadata, not audio files 2026-04-10
    Storing a full audio file per version would be prohibitively expensive. Versions (cutMarks + annotations JSON) are bytes vs megabytes. Source audio stored once; versions are lightweight edit instructions applied at render time or virtually during preview.
  5. 5. Linear versioning only — no simultaneous co-editing 2026-04-10
    Real-time simultaneous editing (CRDT/Y.js) adds significant complexity. The real use case is 3 people each editing a different song. Linear versioning covers all realistic scenarios without conflict resolution logic.
  6. 6. localStorage for in-progress edit safety — no offline sync 2026-04-10
    Full offline support adds complexity. The real need is simpler: if the browser crashes mid-edit, cut marks shouldn't be lost. localStorage solves this with minimal code. Submit still requires a connection — that's acceptable.
  7. 7. Resumable Drive upload from browser — bypass Vercel body limit 2026-04-10
    Vercel's 4.5MB request body limit cannot accommodate a 30-min audio recording (~43MB). Resumable uploads: server generates a Drive upload session URL, client uploads directly to Drive. Audio never passes through Vercel.
Implementation

Stack: Next.js (App Router) + TypeScript + Tailwind CSS, deployed on Vercel Recording: Browser MediaRecorder API + Web Audio API (level meter) + Wake Lock API (screen-on) Waveform editing: wavesurfer.js 7.x (interact:false; cut bands are pure CSS divs — no RegionsPlugin) Storage — audio: Google Drive (resumable upload bypassing Vercel 4.5MB body limit; transcoded to AAC on finalize) Storage — metadata: Neon Postgres free tier (sessions, clips, versions, votes, comments, stamps, bands, members) Auth: NextAuth.js + Google OAuth (drive.file scope) + NextAuth DB adapter (Prisma + Neon, required to store recorder's refresh token for audio proxy) Audio proxy: Next.js API route fetches audio from recorder's Drive using stored token, streams to client with Range support Real-time: Client-side polling every 5-10s when session active Render: fluent-ffmpeg + @ffmpeg-installer/ffmpeg in Next.js API route (triggered on freeze only) Edit safety: localStorage (cut marks saved on every change, restored on editor reopen, cleared on submit) External services: 2 — Google (OAuth + Drive) + Neon (Postgres). Cost: $0/month.

Known Limitations
  • iOS Safari stops MediaRecorder if screen locks or user switches tabs — Wake Lock API (iOS 16.4+) mitigates; warning shown before recording
  • Polling (not WebSocket) — new versions/votes appear within 5-10s, not instantly
  • MediaRecorder output format varies by browser (WebM/Opus on Chrome, AAC/MP4 on iOS) — transcoded to AAC on upload
  • Recorder's Google Drive storage used for band audio — 15GB free, ~350 raw clips before cleanup needed
  • Any Editor-role member can initiate a recording, but all audio lands in the band creator's Drive — one token serves the whole band
Success Criteria
  • Recorder opens browser (any device), records a session, all clips uploaded to Google Drive and metadata saved to Postgres
  • Band member opens a link in mobile browser, opens a clip in the editor within 2 taps
  • Editor loads waveform via Drive audio proxy, marks cuts, previews result without re-encoding
  • Submitting a version writes to Postgres; others see it within 5-10s via polling
  • Three people can each edit a different clip simultaneously, each producing their own version
  • Freeze triggers server-side render, produces downloadable file within 10s for a typical 2-8 min clip
  • Version history shows full chain with author, timestamp, description, and result duration
  • Total infrastructure cost: $0/month
