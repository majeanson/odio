### 🟡 AuthShell — Authenticated Layout Wrapper

`odio-2026-047` · **active** · P1 · `components/layout/AuthShell.tsx`

**Problem:** All authenticated pages need shared layout elements: the BottomTabBar, the UploadBanner (persistent upload progress), and eventually the OfflineBanner (global offline indicator). These cannot live at the page level — they'd need to be duplicated on every route. A single client-side shell mounts them once.

**Analysis:**

AuthShell is a client component mounted in the `app/(auth)/layout.tsx` server layout (which handles auth checks). AuthShell renders:
1. UploadBanner (fixed top, always present when uploads are active)
2. OfflineBanner (fixed top, when offline) — planned
3. Main content slot (`{children}`)
4. BottomTabBar (fixed bottom)

AuthShell extracts `bandId` and `sessionId` from `usePathname()` to pass to BottomTabBar (for Clips tab visibility logic) and to other global components that need routing context.

Does not handle auth redirect itself — that's the responsibility of the server layout using `getServerSession`.

**Implementation:**

components/layout/AuthShell.tsx: 'use client'. Uses usePathname() to extract bandId/sessionId from URL segments. Renders: <UploadBanner />, <OfflineBanner /> (when ready), <div className='flex-1'>{children}</div>, <BottomTabBar bandId={...} sessionId={...} />.

**Key Decisions:**

- **AuthShell is a client component even though most of its content is static**
  > usePathname() requires a client component. Without usePathname(), BottomTabBar cannot determine which tabs to show or hide, and routing context cannot be passed down. The cost of a client boundary here is minimal — AuthShell itself renders no heavy content.

**Known Limitations:**
- bandId/sessionId extraction is regex-based on pathname — URL structure changes require updating the extraction logic

**User Guide:** AuthShell is an internal layout component — users don't interact with it directly. It ensures the upload progress bar and tab navigation are always visible.

### 🟡 BottomTabBar — Four-Tab Persistent Navigation

`odio-2026-046` · **active** · P1 · `components/layout/BottomTabBar.tsx`

**Problem:** The app has four distinct areas (Sessions, Record, Clips, Band) that need to be reachable in one tap from anywhere in the app. Hierarchical-only navigation (back buttons) would require 3+ taps to switch from editing a clip to starting a new recording mid-jam.

**Analysis:**

BottomTabBar renders four tabs fixed at the bottom of the viewport: Sessions (calendar icon), Record (mic icon), Clips (music-note icon), Band (users icon). 56px tall + safe-area-inset-bottom.

Active tab: amber text + icon, label visible. Inactive tabs: text-muted, label hidden on tight viewports.

Hiding rules:
- Tab bar hidden entirely on /edit routes (full-screen editor)
- Record tab hidden for MEMBER-role users (members cannot initiate recording)
- Clips tab hidden when no sessionId context (prevents duplicate tab pointing to same URL as Sessions)

Record tab accent ring pulses amber when a recording is actively in progress (detects via global recording state from Zustand).

**Implementation:**

components/layout/BottomTabBar.tsx: client component. usePathname() to detect /edit routes + extract sessionId. useBandRole() hook for role check. useRecordingStore() for active-recording ring. Renders fixed bottom-0 flex-row nav with 4 items.

**Key Decisions:**

- **Tab bar hidden during editor, not just partially visible**
  > The timeline editor is near-full-screen — the waveform, cut list, and submit button need all available vertical space. A partial tab bar would either cover content or require the editor to shrink. Full hide keeps the editing surface clean.

**Known Limitations:**
- Clips tab disappears when navigating from session to band settings — the sessionId context is not preserved in the URL and is lost
- Tab bar height is hardcoded at 56px — not responsive to font size settings

**User Guide:** The tab bar at the bottom of the screen lets you switch between Sessions, Record, Clips, and Band settings with a single tap from anywhere in the app.

### 🟡 App Shell — Navigation, Dark Theme, Onboarding, Empty States

`odio-2026-004` · **active** · P1 · `components/layout/BottomTabBar.tsx`

**Problem:** The app needs a solid foundation: a responsive web app that works on any device, navigates between recording/editing/collaboration screens fluidly, enforces auth correctly, handles first-time setup (create a band, link a Drive folder) with minimal friction, and never shows a blank screen.

Record tab is hidden for MEMBER-role users. Back navigation uses Next.js <Link> (not <a href>) for client-side transitions — prevents full page reload on every back tap. 2. components/layout/BottomTabBar.tsx: client component — 4 tabs. Hidden on /edit routes via usePathname(). Clips tab hidden when !sessionId (avoids same-URL duplicate). Record tab hidden for MEMBER role via bandMe API query.
3. components/layout/PageLayout.tsx: standard page wrapper. Back button uses Next.js <Link> not <a href> for client-side navigation. Bottom padding: pb-[calc(56px+env(safe-area-inset-bottom))].
4. components/layout/AuthShell.tsx: client shell — mounts BottomTabBar + UploadBanner. Extracts bandId/sessionId from pathname.
5. app/(auth)/layout.tsx: server layout — auth guard, renders AuthShell.
6. components/ui/EmptyState.tsx: reusable empty state with icon, title, description, action.
7. components/ui/Button.tsx, ButtonLink.tsx: shared button components.
8. components/ui/BottomSheet.tsx: slide-up modal for confirmations and input sheets.
9. components/ui/InlineEdit.tsx: in-place editable text field.
10. tailwind.config.ts: dark theme color tokens.

**Key Decisions:**

- **No PWA in V1 — standard responsive web app only**
  > PWA manifest + service worker caching adds complexity without a compelling V1 benefit. The app is used on a browser in the car or at home — bookmarking is sufficient.
- **Dark-only, no light/dark toggle**
  > Primary use context is dim rehearsal rooms and cars at night. Background #0a0a0a, accent amber #f59e0b.
- **Bottom tab bar navigation (not hierarchical-only)**
  > Record needs to be reachable from anywhere in one tap during a live jam. Tab bar hidden only on /edit (full-screen waveform editor).
- **PageLayout back button uses Next.js <Link>, not <a href>**
  > Using <a href> for in-app navigation triggers a full page reload on every back tap — visible flash, loses scroll position, re-runs server components. <Link> performs a client-side transition via the Next.js router.
- **Session page uses pb-[calc(56px+env(safe-area-inset-bottom))] not pb-safe**
  > pb-safe only accounts for the device's safe area inset (home indicator bar). The fixed 56px tab bar sits on top of that. Without adding 56px, the last clip card could be hidden behind the tab bar on short viewports.
- **Hide 'Clips' tab when no sessionId context**
  > When the user is on the band home page or any page without a session context, the Clips tab linked to /bands/[bandId] — identical to the Sessions tab. Having two tabs going to the same URL is confusing and clutters the navigation.
- **System fonts only — no custom font loading**
  > App is used in dark, low-bandwidth settings. Custom fonts add bundle size, FOUT risk, and load time for zero perceptible benefit.

**Known Limitations:**
- No add-to-home-screen in V1 — users must bookmark the app in their browser
- Tab bar not animated — no transition effects between tabs

**User Guide:** After signing in, users land on their bands list. The bottom tab bar (Sessions / Record / Clips / Band) is always visible. Tap Record to go directly to the recording screen from anywhere. The Clips tab appears once you're inside a session — it shows the current session's clips. On first sign-in with no bands, the app redirects to a setup wizard to create the first band.

### ⚪ Toast — Global Feedback Notification System

`odio-2026-019` · **draft** · P1

**Problem:** Multiple screens need to surface transient feedback (upload done, vote saved, copy confirmed, offline detected) without navigating away or blocking the UI. Currently each screen implements its own ad-hoc feedback UI, creating inconsistent timing, placement, and styling.

**Analysis:**

A single shared Toast provider (React context + Zustand or reducer) renders one visible toast at a time from a FIFO queue. Positioned fixed at the bottom of the viewport, 16px above the tab bar (or 16px above safe-area if tab bar is hidden). Max 1 visible at a time — queue drains sequentially.

Three visual types:
- `info`: bg-elevated, no border accent
- `success`: bg-elevated + 4px amber-left border
- `error`: bg-elevated + 4px danger-red left border

Auto-dismiss: info/success after 3s, error after 5s. Manual dismiss × button always present (32px tap target). Text: text-sm text-primary, max 2 lines (no truncation — wraps). No action buttons in toasts — actions live in sheets.

Z-index: above tab bar and all screen content, below modals and bottom sheets. Max width 360px on wide viewports, full-width on mobile.

**Implementation:**

TODO: implement after BottomSheet and layout primitives are stable.

1. lib/toast.ts: `showToast({ message, type, duration? })` — writes to Zustand toastStore. Queue is an array; dequeue on dismiss or timeout.
2. components/ui/Toast/Toast.tsx: client component mounted in AuthShell (same level as UploadBanner). Reads from toastStore. Renders fixed bottom-0 container. Animates in/out with CSS transition (translateY + opacity). Each toast item: bg-elevated div, optional left border, message text, × button.
3. hooks/useToast.ts: helper for components to call showToast without importing the store directly.

**Key Decisions:**

- **No action buttons in toasts**
  > Toasts are purely informational — they confirm that something happened or failed. Actions (Retry, Undo, Confirm) require deliberate user intent and should live in bottom sheets where they have more visual weight and cannot be accidentally tapped during auto-dismiss.
- **FIFO queue, max 1 visible**
  > Stacking toasts creates visual clutter and competes with the tab bar. A queue ensures all toasts are seen without overlapping. FIFO is natural — the first error fired is the most contextually relevant.

**Known Limitations:**
- No persistence — toasts are lost on page reload
- Cannot be triggered from server components — only from client components via showToast()

### 🟡 InlineEdit — Tap-to-Edit Text Field

`odio-2026-024` · **active** · P2 · `components/ui/InlineEdit.tsx`

**Problem:** Session names, clip names, and band name are all editable. Adding a separate 'Edit' icon/button next to each name clutters the UI, especially on small-screen cards. The standard pattern in mobile UIs is tap-the-text-to-edit.

**Analysis:**

InlineEdit renders as text in view mode. On tap: transitions to an `<input>` with the same text styling (no visual reflow — same font size, weight, color). Keyboard opens. Return/Done key saves to Postgres via `onSave(value)` callback (async). Tap away (blur) discards. Escape discards. Character limit enforced via `maxLength`. Empty value is not saved — reverts to previous value.

Used by: session name (max 40ch), clip name (max 60ch), band name (max 40ch).

**Implementation:**

components/ui/InlineEdit.tsx: props { value: string, onSave: (newValue: string) => Promise<void>, maxLength?: number, className?: string }. Internal state: editing (bool) + draft (string). On tap → setEditing(true), autofocus input. On submit/blur: if draft !== value && draft.trim() !== '' → call onSave → setEditing(false) on success. Input styled to match the parent text class via `className` prop passthrough.

**Key Decisions:**

- **Tap away (blur) discards rather than saves**
  > On mobile, the keyboard dismiss gesture triggers a blur event. If blur = save, accidentally dismissing the keyboard would commit an incomplete name. Explicit Return/Done is the save gesture — blur is always a cancel.

**Known Limitations:**
- No multi-line support — clip names limited to single line
- On iOS, tapping away on certain screen areas may not reliably fire blur — user must tap Return to save

**User Guide:** Tap any session name, clip name, or band name to rename it directly. Type the new name and press Done or Return to save. Tap anywhere else to cancel without saving.

_Tags: inline-edit, rename, input, ui-primitive, accessibility_

### ⚪ OfflineBanner — No-Connection State Degradation

`odio-2026-020` · **draft** · P2

**Problem:** When a user loses their network connection mid-session, the app currently fails silently — vote buttons appear interactive but fail on submit, session lists appear to work but show stale data with no indication. This creates confusion and distrust.

**Analysis:**

A fixed top banner mounts in AuthShell and listens to `navigator.onLine` + `online`/`offline` browser events. When offline:

- Banner appears: `bg-elevated`, amber left border, `⚡ No connection — showing last loaded data.` text, full-width, fixed below the system status bar area
- Interactive mutation elements (vote buttons, comment submit, stage chip, submit version) become `disabled` with `aria-disabled` and a `title`/`tooltip`: "You're offline"
- Read-only actions (play audio, browse versions, read comments) remain available if data is loaded
- The Recording Screen is an **exception** — recording itself continues offline; the upload banner handles the upload queue

On reconnection (`online` event):
- Banner auto-dismisses with a brief success state: `✓ Back online` (2s) then disappears
- Polling resumes; queued mutations (if any) retry

**Implementation:**

TODO: implement after Auth Shell layout is stable.

1. components/ui/OfflineBanner/OfflineBanner.tsx: client component. `useOnlineStatus()` hook (see below). Renders fixed top-0 banner when offline. Uses CSS var `--offline-banner-h` (similar to --upload-banner-h pattern) so sticky headers can offset.
2. hooks/useOnlineStatus.ts: `useState(navigator.onLine)` + window event listeners for `online`/`offline`. Returns `{ isOnline, justReconnected }`.

- **Disable mutations, not navigation, while offline**
  > Full app lockdown while offline (preventing navigation) is hostile — the user may be browsing session history or reviewing versions while waiting to reconnect. Only block writes; reads from cached state remain available.

**Known Limitations:**
- navigator.onLine is unreliable in some environments (can report online while behind a captive portal) — OfflineBanner is best-effort
- Queued mutations are not persisted — if the user navigates away and back while offline, pending votes/comments are lost

### 🔒 Auth — Google OAuth via NextAuth.js

`odio-2026-003` · **frozen** · P1 · `lib/auth.ts`

**Problem:** Band members need a shared identity layer — without creating yet another account. Google sign-in is universal in the target audience and doubles as the credential for Google Drive access.

**Analysis:**

NextAuth.js with Google provider. Scopes requested at sign-in: openid, email, profile, https://www.googleapis.com/auth/drive.file (access only to files Odio creates in the user's Drive). No Supabase. The access_token and refresh_token are stored server-side in the NextAuth session (encrypted cookie or database adapter). All Drive API calls are proxied through Next.js API routes — the token is never exposed to the client. Auth state (user email, displayName, avatar) held in Zustand. The public frozen clip page (/share/[token]) is the only unauthenticated route.

**Implementation:**

1. lib/auth.ts: NextAuth.js config with GoogleProvider (scopes: openid + email + profile + drive.file). PrismaAdapter with Neon Postgres. Session strategy: database.
2. app/api/auth/[...nextauth]/route.ts: standard NextAuth route handler.
3. middleware.ts: protects all routes except /, /login, /share/[token]. Redirects unauthenticated requests to /login.
4. lib/prisma.ts: singleton Prisma client.
5. Prisma schema: User, Account (NextAuth tables), Band, BandMember, JamSession, Clip, ClipVersion, Vote, Comment tables.
6. app/(auth) route group: all authenticated pages wrapped in a layout that verifies session.
7. app/login/page.tsx: sign-in page with Google OAuth button.
8. The accounts table managed by PrismaAdapter stores access_token + refresh_token per user — required for server-side Drive API calls.

**Key Decisions:**

- **drive.file scope at sign-in, not deferred**
  > drive.file is needed for all core features (upload recording, read session/clip data, write meta). Deferring it to a later step would require a second OAuth flow and would break the first recording attempt. Request it upfront with a clear explanation in the sign-in screen: 'Odio only accesses files it creates in your Drive.'
- **NextAuth.js with Prisma DB adapter — required from day one, not deferred**
  > Supabase Auth adds a paid service. NextAuth.js is open-source and integrates directly with Next.js. The DB adapter (Prisma + Neon) is not optional: it is required to persist the recorder's refresh_token in the accounts table, which is the only way the audio proxy route can stream Drive audio to all band members. JWT-only session strategy is explicitly rejected — it cannot store the refresh_token server-side.

**Known Limitations:**
- drive.file scope only covers files the app created under that specific user's account — bandmates stream audio through the server-side proxy (which uses the recorder's stored token), so bandmates do not need Drive scope just to listen
- Google OAuth tokens expire after 1 hour — NextAuth DB adapter stores refresh_token in Postgres and rotates on expiry; JWT-only strategy is insufficient and must not be used
- If the recorder revokes Odio's Drive access (in Google account settings), all audio becomes inaccessible to the entire band until re-authorized

**User Guide:** Users visit the app and click 'Sign in with Google'. They are prompted to grant Google Drive access (files created by Odio only). After sign-in, they are redirected to the bands home page. Sessions persist across browser restarts. Sign-out via the Band settings page. The /share/[token] public page requires no sign-in.

### 🟡 StampRow — One-Tap Timestamp Annotation During Recording

`odio-2026-028` · **active** · P1 · `components/recording/StampRow.tsx`

**Problem:** During a live recording, a band member wants to mark interesting moments (a great riff, an uncertain section, an idea) without interrupting the take. Any UI with confirmation dialogs or text input is too slow and disruptive mid-jam.

**Analysis:**

Four large buttons equally spaced across the recording screen: 🔥 (fire/hot, stamp-fire red), ⭐ (keep, stamp-keep yellow), ❓ (uncertain, stamp-uncertain orange), 💡 (idea, stamp-idea blue). Each button is 72px minimum tap target. One tap:
1. Triggers light haptic feedback
2. Records the current `recordingElapsedMs` as a timestamp
3. Saves a `{ stampType, timestampMs }` to local state (flushed to Postgres on recording stop)

No confirmation, no text, no visual feedback beyond a brief button scale animation. Zero friction is the goal.

First-use: the StampHint component (sibling) shows labels beneath the buttons for 5s on first visit.

**Implementation:**

components/recording/StampRow.tsx: props { onStamp: (type: StampType, timestampMs: number) => void }. StampType = 'FIRE' | 'KEEP' | 'UNCERTAIN' | 'IDEA'. Renders 4 buttons in flex-row gap-4. Each button: rounded-full, 72px, emoji centered, colored ring. onPointerDown (not onClick) fires for minimum tap latency. Scale animation via CSS class toggle.

**Key Decisions:**

- **onPointerDown instead of onClick for minimum latency**
  > onClick fires after both pointerdown + pointerup (adds ~100-150ms vs pointerdown). For a live musical timestamp, 100ms earlier capture is meaningful — the beat has moved on. StampType annotation is written on pointerdown.
- **Stamps flushed to Postgres on recording stop, not per-stamp**
  > One Postgres write per stamp during recording creates N roundtrips mid-take, each competing with the audio recording thread. Batch write on stop: one POST /api/clips/[clipId]/stamps with all stamps collected during the session.

**Known Limitations:**
- navigator.vibrate() not supported on iOS — no haptic on Apple devices
- Stamp timestamps are relative to recording start (elapsedMs) — if recording is paused (e.g., wake lock interruption), timestamps after the pause will be offset

**User Guide:** Tap any of the four icons while recording to mark that moment — 🔥 for something hot, ⭐ to flag a keeper, ❓ for uncertainty, 💡 for an idea. Your marks appear as colored dots on the waveform later.

### 🔒 Recording — Browser-Based Audio Capture

`odio-2026-006` · **frozen** · P1 · `hooks/useRecorder.ts`

**Problem:** The recorder needs to capture multiple songs during a live jam — from a phone browser, one hand, dim room. Between songs the transition to the next recording must be instant. Moments worth keeping must be stampable without interrupting the performance.

**Analysis:**

Browser MediaRecorder API with getUserMedia. Typical clip: 5-30 min raw. Audio buffered in memory during recording (30 min AAC ~43MB — fits easily). On stop: upload to Google Drive via resumable upload (server generates upload session URL, browser uploads directly to Drive, bypassing Vercel 4.5MB body limit). Server then transcodes to AAC via FFmpeg (cross-browser compatibility — Chrome outputs WebM, Safari outputs AAC/MP4). Transcoded file replaces the original in Drive. Clip and v1 version rows created in Postgres. Wake Lock API active during recording (screen-on). Level meter via Web Audio API AnalyserNode. Stamps (fire/keep/uncertain/idea) buffered in memory, written to Postgres with clip on stop. 'Next Song' button appears after stop: saves current clip + immediately opens new recording, upload continues in background.

**Implementation:**

1. app/record/page.tsx: full-screen recording screen — timer, level meter, stamp row, Add note link, record/stop button. Tab bar hidden via usePathname().
2. hooks/useRecorder.ts: core hook managing MediaRecorder lifecycle, Web Audio AnalyserNode for level meter, Wake Lock API, stamps buffer, annotations buffer, device enumeration and selection, IndexedDB write on stop (via savePendingUpload), upload start.
3. components/recording/DevicePicker.tsx: bottom sheet listing available audio input devices.
4. app/record/post/page.tsx: post-record screen — clip name (inline edit), duration, stamp count, upload progress bar, Next Song button, Delete recording button.
5. hooks/useUploadProgress.ts: polls /api/clips/[clipId] every 3s until transcodeStatus is DONE or FAILED.
6. lib/pendingUploads.ts: IndexedDB helpers for pending upload persistence.
7. app/api/upload/init/route.ts: generates Drive resumable upload session URL using creator's stored token.
8. app/api/upload/finalize/route.ts: transcodes uploaded audio with FFmpeg, creates Clip + ClipVersion rows in Postgres.
9. STAMP_EMOJI and STAMP_COLORS constants in types/index.ts.

**Key Decisions:**

- **Resumable Drive upload from browser (bypasses Vercel body limit)**
  > Vercel serverless functions have a 4.5MB request body limit. A 30-min recording at 192kbps ~43MB. Sending audio through Vercel would fail. Resumable uploads: server generates a Drive upload session URL, browser POSTs audio blob directly to Drive. Audio never passes through Vercel.
- **Transcode to AAC immediately on upload**
  > MediaRecorder outputs WebM/Opus on Chrome and AAC/MP4 on Safari. WebM does not play in Safari. Transcoding immediately on upload ensures one consistent format (AAC) for all playback, editing, and final render — regardless of which browser the recorder used.
- **Any Editor-role member can record — not just the band creator**
  > The upload flow already uses the creator's stored refresh_token on the server to generate the Drive resumable upload session URL. The member's browser uploads directly to the creator's Drive. This means multi-recorder support is not an architectural change — it's a role-permission change. Any Editor can initiate a recording and audio lands in the same Drive folder. Reduces the single-point-of-failure where only one person can capture audio on jam night.
- **Two notation types: stamps (typed, no text) and annotations (text notes) — both timestamped**
  > Stamps are one-tap typed markers — fire/keep/uncertain/idea — zero friction during playing. Annotations are short text notes for when you want to say something specific. Both appear as pins on the waveform. Keeping them separate preserves the zero-friction stamp UX while allowing richer notes when needed. They serve similar purposes at different levels of effort.

**Known Limitations:**
- iOS Safari stops MediaRecorder if the screen locks or the user switches tabs — Wake Lock API (iOS 16.4+) mitigates accidental sleep; tab switching is user responsibility
- If the browser tab is closed before the blob is written to IndexedDB (i.e., during the first few seconds after stop), the audio may be lost — mitigated by writing to IndexedDB immediately on stop before any upload attempt (see feat-recording-resilience)
- Resumable upload requires two API round-trips before recording is saved — adds ~2-4s to the stop flow
- Any Editor-role member can initiate a recording, but all audio lands in the band creator's Drive — members upload directly to the creator's Drive using the server-generated resumable upload session URL (which uses the creator's stored token)

**User Guide:** Tap the Record tab from anywhere. On the recording screen, tap the large circle button to start. The timer runs; the level meter shows audio input. Tap stamp buttons (fire/keep/uncertain/idea) any time to mark notable moments. Tap 'Add note' to type a quick text annotation. Tap the square stop button to finish. After stopping, name the clip on the post-record screen, tap 'Next Song' to immediately start another recording while the previous one uploads in the background, or navigate away.

### 🔒 Multi-Recorder — Any Editor Can Record Into the Band's Drive

`odio-2026-008` · **frozen** · P2 · `app/api/upload/init/route.ts`

**Problem:** On jam night, the person who created the band may not always be the one with a phone free to record. Any Editor-role band member should be able to hit Record and have the audio land in the band's shared Drive — without requiring each member to set up their own Drive folder or the band creator to be physically present.

**Analysis:**

The existing recording upload flow already makes multi-recorder architecturally trivial: when a member hits Record, the server's /api/upload/init route generates a Google Drive resumable upload session URL using the band creator's stored refresh_token (saved via NextAuth DB adapter). The member's browser then uploads the audio blob directly to the creator's Drive — the member's own Drive is never involved. The audio lands in the correct band Drive folder regardless of who recorded it.

What needs to change is a role-permission gate: currently /api/upload/init checks that the caller is the band creator (Recorder role). Expanding this to allow Editor-role members is the full scope of the server-side change.

On the client: the Record tab in the bottom nav is shown only to Recorder and Editor roles. Member-only users do not see the Record tab.

One critical edge case: the creator's Drive token can expire. The server must attempt a token refresh before generating the upload session URL. If the refresh fails (e.g., creator revoked app access), the error must be surfaced clearly — 'Band creator needs to re-authorize Drive access' — with a notification mechanism to alert the creator.

The resulting clip row in Postgres records both the band creator (driveOwnerEmail) and the actual recording member (recordedByEmail) so the activity feed can correctly show 'Recorded by [member]'.

**Implementation:**

1. app/api/upload/init/route.ts: POST — role check `['RECORDER', 'EDITOR'].includes(membership.role)`. Fetches band creator's email from bands table. Queries NextAuth accounts table for creator's access_token + refresh_token. Attempts token refresh via Google OAuth token endpoint if expires_at < now. Generates Drive resumable upload session URL using creator's token. Returns { uploadUrl, clipId, sessionId }.
2. app/api/upload/finalize/route.ts: POST — same role check. Runs FFmpeg transcode on uploaded file. Creates Clip row with recordedByEmail = currentUser.email. Creates v1 ClipVersion row. Updates clip transcodeStatus to DONE.
3. Error handling: if creator token refresh fails, returns 503 with { error: 'CREATOR_TOKEN_INVALID', creatorEmail }. Client shows 'Drive connection needs renewal — ask [creator name] to re-authorize'.
4. components/layout/BottomTabBar.tsx: Record tab hidden when `membership.role === 'MEMBER'`.
5. Prisma schema Clip: recordedByEmail TEXT nullable. clips table stores both fields.
6. components/clips/ClipCard.tsx: shows 'Recorded by X' when recordedByEmail differs from band creator.

**Key Decisions:**

- **Reuse creator's stored token — no per-member Drive setup**
  > drive.file OAuth scope only grants access to files the app created under that specific user's account. A member's token cannot write to the creator's Drive folder. The creator's refresh_token (already stored by NextAuth DB adapter for the audio proxy feature) solves this: the server uses the creator's token to generate the Drive upload session URL, and the member's browser uploads directly to Drive using that URL. No additional OAuth consent or per-member Drive setup needed.
- **Editor role can record — Member role cannot**
  > Member-only users are passive collaborators (vote, comment). Giving them recording rights would require explaining Drive ownership semantics to every casual invitee. Editors are already trusted with version creation and freeze — recording is a natural extension. The default role for new band invites is Editor, so most band members get recording rights automatically without any explicit configuration.
- **Track recordedByEmail separately from driveOwnerEmail**
  > The clip always lives in the creator's Drive (driveOwnerEmail is always the band creator). The person who pressed Record may be someone else entirely. These are distinct facts: one describes file ownership for Drive API operations, the other describes recording attribution for the activity feed and session history. Conflating them would make it impossible to correctly display 'Recorded by X' or debug Drive issues per-clip.

**Known Limitations:**
- If the band creator revokes Google Drive access, ALL recording attempts by any Editor fail until the creator re-authorizes — one token serves the whole band; there is no per-member fallback
- Two Editors recording simultaneously produce two separate clip rows with overlapping timestamps — no merging or conflict detection; both appear in the session normally
- The creator's refresh_token is stored server-side in the NextAuth accounts table — this security surface already exists for the audio proxy feature; multi-recorder adds no new exposure

**User Guide:** Any Editor-role band member can record — not just the band creator. Editors see the Record tab in the nav and can start a recording from their own device. The audio automatically lands in the band's Drive folder (owned by the band creator). If the creator's Drive connection expires, a banner appears: 'Drive connection needs renewal — ask [creator name] to re-authorize.' The clip card shows 'Recorded by X' when the recorder is not the band creator.

_Tags: recording, multi-recorder, roles, permissions, google-drive, upload_

### 🟡 Upload Banner — Persistent Upload Progress Across All Pages

`odio-2026-010` · **active** · P2 · `components/common/UploadBanner.tsx`

**Problem:** After stopping a recording, the band member needs to navigate away (to another song, to the session list, to collaborate) without losing visibility into whether the upload succeeded. The upload progress must persist across all pages — not just the record screen — as a fixed top banner.

**Analysis:**

UploadBanner.tsx is a fixed-position (top-0, z-50) client component mounted in AuthShell, making it visible on every authenticated page. It receives the list of active pending uploads from usePendingUploads and renders the most actionable one prominently.

Status priority order: token-error > session-error > paused > uploading. The most actionable upload is shown first — an error that requires human action is more urgent than an active upload or a paused one.

Three visual variants:
- **uploading**: amber background, spinner icon, progress bar below text (byteOffset / blob.size)
- **paused**: elevated background, no spinner, Retry button
- **error**: danger/10 background, red text, Retry + 'Save file' buttons

Progress bar is shown for active uploads only, using byteOffset from IndexedDB to compute percentage. When multiple uploads are pending, the banner shows the primary one with '+N more' count.

The banner auto-dismisses (returns null) when uploads.length === 0 — no manual close button for active uploads, since closing it would hide critical progress state.

**Implementation:**

1. components/common/UploadBanner.tsx: client component. Fixed position top-0, full-width, z-50. On desktop (md+) offset right to avoid sidebar: md:left-[220px]. Progress percentage computed as a dedicated numeric variable (not inline JSX short-circuit, which renders literal '0' when byteOffset is 0): progressPercent = blob.size > 0 && byteOffset != null ? Math.round(byteOffset / blob.size * 100) : 0. Progress bar is absolute bottom-0 (2px strip, stable height). Error states (token-error, session-error) include a dismiss ✕ button — active/paused uploads do not. Uses ResizeObserver to write its height to --upload-banner-h CSS variable so sticky headers below can offset correctly. Returns null when uploads.length === 0.
2. components/layout/AuthShell.tsx: mounts UploadBanner + BottomTabBar at auth layout level. Passes upload state from usePendingUploads hook.
3. hooks/usePendingUploads.ts: reads IndexedDB on mount, drives retry state machine, exposes uploads[], retry/discard/saveToDevice handlers to AuthShell.
4. app/record/post/page.tsx: PostRecordScreen.doUpload() calls usePendingUploadsStore.getState().refresh() on every status-changing path (success, session-error, token-error, paused, catch) so the Zustand store stays in sync with IndexedDB and the banner dismisses immediately after upload completes without requiring a page refresh.

**Key Decisions:**

- **Most actionable upload shown first — errors before paused before active**
  > A token error requires a human action (band creator re-authorizes). A session error requires sign-in. These block recovery and are more urgent than an active upload silently progressing. Showing errors first ensures the user notices and acts on the blocker rather than assuming uploads are fine.
- **No close button on active/paused uploads; dismiss button only on error states**
  > Closing an active or paused upload banner hides critical progress state and gives false confidence. But permanently blocking the UI for an unresolvable error (token-error, session-error) that the user cannot fix immediately is hostile — they may need to navigate while waiting for the creator to re-authorize. Error states have an ✕ dismiss button that hides the banner without discarding the upload (it can still be retried from any page).

**Known Limitations:**
- Banner occupies top space — sticky headers below must consume --upload-banner-h CSS variable; new pages with their own sticky headers need to adopt this pattern
- On desktop the banner is offset md:left-[220px] to avoid the sidebar; if the sidebar width changes, this value must be updated manually
- Progress bar accuracy depends on IndexedDB byteOffset being updated frequently during upload; if updates are sparse, the bar may jump
- 'Save file' downloads the raw browser-recorded blob (WebM on Chrome, AAC/MP4 on Safari) — not the transcoded AAC that would normally be stored in Drive

**User Guide:** After stopping a recording, a banner appears at the top of every screen showing upload progress. If the upload completes successfully, the banner disappears automatically. If the connection drops, the banner shows 'Upload paused' with a Retry button — tap it once reconnected. If the band creator's Drive connection expires, the banner shows an error with a 'Save file' button as a last resort to download the raw recording to your device.

### 🔒 Recording Resilience — IndexedDB Safety Net, Upload Retry, Save-to-Device Fallback

`odio-2026-009` · **frozen** · P2 · `hooks/usePendingUploads.ts`

**Problem:** A recording is a one-time, irreplaceable event. Currently, if the internet drops during upload or the band creator's Drive token is invalid, the audio blob — which only exists in memory — is permanently lost when the user navigates away or the tab closes. A live jam session cannot be re-played. The recording must survive any upload failure and remain recoverable until it reaches Drive.

**Analysis:**

The audio blob lives in RAM from record-start until upload completes. If anything breaks during the upload window — network drop, Drive token expiry, browser crash, tab close — the blob is gone. This is the highest-risk data loss scenario in the app.

The fix is a two-layer safety net:

Layer 1 — IndexedDB persistence (immediate): The moment recording stops and the blob is assembled, write it to IndexedDB before starting the upload. IndexedDB stores binary blobs up to the device's available storage (typically several GB on modern phones), survives page refresh and browser restarts, and is cleared only explicitly. Key: odio-pending-upload-{tempId}. Also store: tempId, recordedAt, durationMs, bandId, sessionId, blob.

Layer 2 — Resumable upload retry: Google Drive resumable uploads natively support resume from a byte offset. If the upload is interrupted mid-transfer, the server stores the upload session URL in the pending record. On retry, the client queries Drive for the current byte offset and resumes from there — no re-upload of already-transferred bytes.

**Implementation:**

1. lib/pendingUploads.ts: IndexedDB helpers using idb library. Database 'odio-recordings', object store 'pending-uploads'. Functions: savePendingUpload(upload), getPendingUploads(), deletePendingUpload(tempId), updatePendingUpload(tempId, patch).
2. hooks/useRecorder.ts: calls savePendingUpload immediately on MediaRecorder onstop before any network request.
3. hooks/usePendingUploads.ts: checks IndexedDB on mount, drives retry logic, manages upload state machine (idle/uploading/paused/error), listens to window 'online' event for auto-retry.
4. components/recording/PendingUploadsBanner.tsx: recovery banner shown when usePendingUploads reports pending items. Shows count, retry button, save-to-device button, discard button with confirmation.
5. app/(auth)/layout.tsx: renders PendingUploadsBanner at top of all authenticated pages.
6. lib/uploadAudio.ts: manages the Drive resumable upload from browser — tracks byte offset, handles chunking, updates IndexedDB with uploadSessionUrl on each successful chunk.
7. 'Save to device': URL.createObjectURL(blob) + programmatic anchor download. Extension .aac or .webm based on mimeType.

**Key Decisions:**

- **IndexedDB as the primary persistence layer for unuploaded audio blobs**
  > localStorage cannot store binary blobs. The File System Access API requires explicit user permission. IndexedDB is the only browser-native store that (a) handles large binary data, (b) persists across sessions without user action, and (c) is available in all browsers including iOS Safari 10.3+. A 43MB blob fits easily — IndexedDB storage limits are typically 50-80% of available disk space.
- **Resumable upload — save session URL to IndexedDB for byte-accurate retry**
  > A naive retry re-uploads the entire blob from byte 0. For a 43MB file on a slow mobile connection, this could fail again at the same spot. Google Drive resumable uploads let you query the committed offset and resume from there. Storing the uploadSessionUrl in IndexedDB alongside the blob enables true resume, not restart.
- **'Save to device' as the last-resort fallback — never lose the audio**
  > If Drive is permanently unavailable (creator deleted their account, Drive quota exceeded, token irrecoverable), the band should still get their recording. A browser download gives them the raw file. They lose the Odio versioning workflow but not the audio. The guarantee is: if you recorded it, you can get the file.
- **Check IndexedDB for pending recordings on every app mount**

- PostRecordScreen.doUpload() is a standalone async function that writes to IndexedDB directly without going through the Zustand store's action methods — fixed by calling usePendingUploadsStore.getState().refresh() on every status-changing path so the in-memory store stays in sync with IndexedDB and the banner dismisses immediately

These nodes intercept pointer events even with pointerEvents:none on individual regions because the RegionsPlugin wrapper div captures them first. Pure CSS absolute-positioned divs with pointer-events:none do not interfere with the overlay div at all, and their position can be computed from millisecond values without any WaveSurfer API calls.
  > _Alternatives considered: WaveSurfer RegionsPlugin — conflicts with custom overlay, harder to scroll-correct positions, Canvas overlay — requires repainting on every animation frame; CSS divs reflow only on prop change_
- **Stamp chips in WaveformPlayer/WaveformEditor/PublicPlayer are tappable buttons via StampJumpRow**
  > Divs are not keyboard-accessible and do not receive touch events reliably on all browsers. Using <button> elements makes stamp chips proper interactive controls. StampJumpRow is a shared component used by all three players — one source of truth for chip styling and seek behavior.
- **Split navigates to session page, not back to clip detail**
  > After a split, two clips exist. router.back() would go back to the original clip's detail page — the user would not see Part B at all without manually navigating. router.push(sessionHref) lands on the session clip list where both Part A and Part B are visible and accessible.
- **Virtual cut preview via seek — no re-encode**
  > Re-encoding for preview requires server round-trip or WASM FFmpeg. WaveSurfer can seek to any position — skip cut regions during playback by intercepting timeupdate events in useWaveSurfer. Zero server cost, instant response.
- **Drive audio proxied through Next.js API route**
  > Drive API download URLs require an Authorization header with the user's access token. Browsers cannot add auth headers to an <audio> src or WaveSurfer source URL. Proxying through a Next.js API route (which has the token server-side) solves this without exposing the token to the client.
- **localStorage for in-progress edit safety**
  > Full offline sync adds complexity. The real risk is a browser crash mid-edit. localStorage persists cut marks across crashes. Key: odio:draft:{clipId}. Cleared on successful submit.

**Known Limitations:**
- Audio proxy route adds latency vs direct URL — mitigated by the server having fast egress to Google's CDN
- If the clip's Drive file is unavailable (recorder revoked share, Drive outage), editor shows an error with Reconnect Drive option
- Zoom level resets to 1x when entering split mode — by design, split mode always shows the full waveform for context

**User Guide:** From a clip's detail page, the waveform player shows cuts as dark overlays. Tap any stamp chip to jump to that moment. Click-to-seek works anywhere on the waveform — even on cut overlays. Tap 'Edit cuts' to open the full-screen editor. In the editor: drag on the waveform to create a red cut region; drag the edge of an existing region to resize it; tap Trim to pre-seed start/end cuts; tap Clear all to reset; tap Split to divide the song into two clips at a chosen time position. Tap 'Start from vN' to load a previous version's cuts as a starting point. Tap 'Submit edit' to save as a new version — description is optional. Two dedicated components with separate concerns:

Big 64px play button. finish event resets cursor to start (ws.setTime(0)) so the amber cursor doesn't pin at the right edge after playback ends.

Submit → new version in Postgres, router.back().

**Non-destructive split with immediate Drive files**: original clip is never mutated. Splitting Song X into Part A and Part B: the split API downloads X's source once, runs FFmpeg twice in parallel to produce two trimmed files (A: 0→splitMs, B: splitMs→end), uploads both to Drive with their own names, and creates two new Clip rows with their own driveFileId + correct sourceDurationMs + empty v1 cutMarks (the trimmed file is their raw). The original clip is untouched. After confirming that A and B are good, X's source can be deleted from Drive to free space. No provenance UI is shown on split children (no scissors icon, no 'Split into' card) — they are fully independent songs from the user's perspective. sourceClipId exists in the DB for data integrity but is not surfaced in the UI.

**ClipDetailClient** — owns selectedVersionId state. Feeds activeCuts to WaveformPlayer. Feeds activeVersionId to CollaborationSection for version-aware voting. Version list as large tappable cards. Clicking active version keeps it selected (no deselect/null state). Edit link to /edit page.

3. components/clips/ClipDetailClient.tsx: coordinator. Owns selectedVersionId state (auto-selects latest, no deselect). Passes activeCuts to WaveformPlayer. Passes activeVersionId to CollaborationSection. Actions section hidden when !canEdit && !frozen.

6. app/api/clips/[clipId]/versions/route.ts: POST creates a new version.

**Key Decisions:**

- **WaveformPlayer and WaveformEditor are separate components**
  > A single component that handles both playback and editing requires mode-switching logic, conditional overlays, and competing event handlers. Separate components have clear single responsibilities: Player = listen, Editor = cut. The edit page is always in edit mode; no toggle needed. The detail page never needs editing. Each component can be reasoned about and tested independently.
- **interact:false + single overlay div for all pointer handling (no enableDragSelection, no RegionsPlugin)**
  > WaveSurfer's enableDragSelection registers a permanent pointer overlay that consumes all pointer events — including click-to-seek. Once called, seeking by clicking is broken. interact:false disables WaveSurfer's built-in event handling entirely. A single overlay div (absolute inset-0, touch-none) inside WaveformCanvas intercepts all pointer events and forwards them to useCutInteraction (trim mode) or useSplitInteraction (split mode).
- **Split produces independent trimmed Drive files immediately (not virtual clips sharing source)**
  > Original model: Part A and Part B shared Song X's Drive file and applied cuts virtually — correct waveform, wrong duration in clip list (both showed full 9-min duration instead of 4 min and 5 min). Trimmed files fix duration display without special-casing. They also make Part A and Part B genuinely independent: they can be frozen separately without needing X's file, and X's source can be safely deleted from Drive once A and B are confirmed. FFmpeg at split time costs a few seconds of processing but eliminates a class of edge cases.
- **No provenance UI on split children**
  > Showing scissors icons and a 'Split into' provenance card on Song X implied that Song X was consumed or changed by the split. The correct mental model is: X exists unchanged, A and B are new songs. Their name might hint at their origin (e.g. 'Groove - Part A') but they are first-class independent songs from the moment they're created. Provenance exists in the DB (sourceClipId) for integrity but is not surfaced in the UI.
- **Split navigates to session page, not back to clip detail**
  > After a split, three clips exist (original + Part A + Part B). router.back() would return to the original clip's detail — neither Part A nor Part B would be visible. router.push(sessionHref) lands on the session clip list where all three are visible and accessible.
- **No version deselect — clicking active version keeps it selected**
  > Deselecting a version set selectedVersionId to null, which caused the player to show 'no cuts' (raw audio). This 'null = raw audio' state had no UI label and was visually identical to a version with no cuts. Confusing. Clicking the active version now keeps it selected — the player always reflects a known version.
- **Versions are metadata only — not audio files**
  > Audio duplication per version is prohibitively expensive (43MB+ per clip). Versions are cut instructions (cutMarks JSON). Source audio stored once in Drive. Real audio file produced only on freeze.
- **localStorage for in-progress edit safety**
  > Full offline sync adds complexity. The real risk is a browser crash mid-edit. localStorage persists cut marks across crashes. Key: odio:draft:{clipId}. Cleared on successful submit.

**Known Limitations:**
- Split takes a few seconds on the server (FFmpeg download + trim + upload) — the UI shows a loading state on the confirm button but gives no progress indicator
- After splitting, the original Song X remains in the session — the band must manually delete it from the Drive page once they've confirmed A and B are good

- Submit requires internet connection — cuts safe in localStorage but cannot be written to Postgres offline

### 🔒 FFmpeg POC — Validate Bundle Size and Render Pipeline on Vercel

`odio-2026-002` · **frozen** · P1 · `app/api/render/poc/route.ts`

**Problem:** @ffmpeg-installer/ffmpeg bundles the FFmpeg binary at ~38MB compressed. Vercel's serverless function size limit is ~50MB (250MB uncompressed). This is the single highest-risk technical constraint in the project. If the bundle is too large or the render takes too long (Vercel function timeout: 10s on Hobby), the entire render architecture needs to change before feat-render can be built. This POC must validate or invalidate the approach before any production code is written.

**Analysis:**

The POC validates three things in isolation:

1. **Bundle size:** Does a Next.js API route that imports @ffmpeg-installer/ffmpeg + fluent-ffmpeg deploy successfully to Vercel Hobby without hitting the 50MB compressed / 250MB uncompressed limit? The binary alone is ~38MB compressed — leaving ~12MB for the rest of the function bundle.

2. **Cold start:** Vercel serverless functions have a cold start on first invocation. A 38MB binary means a large cold start. Is the first render under 30 seconds? Subsequent renders (warm function) should be under 10s for a typical 2–8 min clip output.

3. **Render correctness:** Does a fluent-ffmpeg atrim+concat command correctly produce a trimmed AAC file from a source AAC file on Vercel's runtime environment?

If bundle size fails: alternatives are (a) Vercel Pro (1GB function size limit, $20/month — viable), (b) a dedicated render worker (Railway, Fly.io free tier), (c) client-side WASM FFmpeg (slow on mobile, acceptable for desktop-only), or (d) a cloud transcoding service (AWS MediaConvert — adds cost and complexity).

If timeout fails: the 10s Vercel Hobby limit may be too tight for longer clips. Vercel Pro raises the limit to 300s. The POC should test with a worst-case 30-min source clip.

The POC is a standalone Next.js route — not integrated into the app — that accepts a test audio file and returns the trimmed output.

**Implementation:**

1. Created GET /api/render/poc route that accepts ?url= param pointing to a source audio file.
2. Installed @ffmpeg-installer/ffmpeg + fluent-ffmpeg as production dependencies.
3. Route downloads source audio to /tmp, runs fluent-ffmpeg atrim+aconcat filter with a hard-coded test cut (0-10s removed), outputs trimmed .aac.
4. Returns rendered file as application/octet-stream response.
5. Result: bundle deploys on Vercel Hobby within size limit. Cold start ~8s, warm render ~3s for a 5-min clip. Architecture validated.
ComponentFile: app/api/render/poc/route.ts

**Key Decisions:**

- **POC before any production render code**
  > @ffmpeg-installer/ffmpeg is the highest-risk dependency in the entire project. If it fails on Vercel Hobby, the render feature needs a completely different architecture. Building feat-render on top of an unvalidated assumption would waste significant implementation work. The POC is a 2-4 hour investment that prevents a potential full architectural rethink later.

**Known Limitations:**
- Vercel Hobby function timeout is 10s — this may be too tight for long clips; Pro plan raises to 300s
- If bundle size fails on Hobby, Vercel Pro ($20/month) is the simplest fix — breaks the $0/month cost target

**User Guide:** This is an internal validation endpoint, not a user-facing feature. To test: GET /api/render/poc?url=<public-audio-url>. The endpoint downloads the audio, applies a test cut (removes seconds 0-10), and returns the trimmed file. Used during development to confirm Vercel can run FFmpeg before building the real render pipeline.

### 🔒 Render — Server-Side FFmpeg on Freeze + Public Share

`odio-2026-014` · **frozen** · P2 · `app/api/clips/[clipId]/freeze/route.ts`

**Problem:** When a version is frozen, a real audio file must be produced — so the band can download it, share it outside the band, or export it. This cannot happen client-side. The frozen clip should also be shareable publicly without requiring sign-in.

**Analysis:**

Next.js API route POST /api/render: fetch source .aac from Drive (via recorder's stored token), run FFmpeg atrim+concat to apply cutMarks, output final .aac at 192kbps, upload result as {clipId}-final.aac to Drive. Update clips row: frozen=true, frozenVersionId, frozenDriveFileId. Progress communicated via Postgres poll — client polls GET /api/clips/{id} every 3s until frozen=true. For public share: frozen clips get a public Drive permission set ('anyone with link can read') and a short public token stored in Postgres. Public page /share/[token] fetches clip metadata from Postgres and streams audio from the Drive public URL — no sign-in required.

**Implementation:**

1. app/api/clips/[clipId]/freeze/route.ts: POST — Editor+ role. Validates versionId. Fetches creator's Drive token from NextAuth accounts table. Downloads source .aac to /tmp. Runs fluent-ffmpeg atrim+aconcat with cutMarks from the version. Uploads output as {clipId}-final.aac to band's Drive folder. Sets clip.frozen=true, clip.frozenVersionId, clip.finalDriveFileId. Returns updated clip.
2. app/api/clips/[clipId]/unfreeze/route.ts: POST — RECORDER only. Sets clip.frozen=false, clip.frozenVersionId=null. Keeps finalDriveFileId.
3. app/api/clips/[clipId]/route.ts: PATCH handles generatePublicToken (requires frozen=true — sets publicToken=crypto.randomUUID(), sets Drive public permission) and removePublicToken (clears publicToken, removes Drive public permission).
4. app/share/[token]/page.tsx: public page — no auth. Fetches clip by publicToken from Postgres. Renders waveform player with Drive public audio URL + clip name + stamps.
5. FFmpeg atrim+aconcat filter chain: for each non-cut segment, atrim=start=X:end=Y, asetpts=PTS-STARTPTS, then aconcat=n=N. Output: -b:a 192k -f adts (AAC).
6. lib/drive.ts: downloadToTemp(fileId, token) → /tmp/{fileId}.aac. uploadFromTemp(path, folderId, token) → new driveFileId. setPublicPermission(fileId, token). removePublicPermission(fileId, permissionId, token).

**Key Decisions:**

- **Server-side FFmpeg only — no client-side WASM**
  > WASM FFmpeg is slow on mobile (~3x slower than native), memory-constrained, and adds bundle size. Server-side runs once per clip lifetime in < 10s on Vercel. Acceptable trade-off.
- **Public share via Drive public permission + Postgres token**
  > A frozen clip can be shared outside the band (with a producer, a friend). Setting Drive file permission to 'anyone with link' gives a direct audio URL. Odio stores a short token in Postgres mapping to the clip, so the public URL is /share/[token] (not a raw Drive ID), giving control over revocation.

**Known Limitations:**
- Vercel /tmp is ephemeral (512MB limit) — source download (~43MB) + FFmpeg output (~10MB) is within limits for typical clips
- Render requires recorder's valid Drive token from Postgres — if token refresh fails (e.g., recorder revoked access), render fails with a clear error and re-auth prompt
- @ffmpeg-installer/ffmpeg bundles ~38MB compressed — close to Vercel's function size limit; this is the highest-risk technical component and needs a spike before building
- Any band Editor role can trigger freeze, but the render always uses the recorder's Drive token — recorder does not need to be signed in at the moment of freeze

**User Guide:** On a clip's detail page, tap 'Freeze' on any version to produce the final audio file. The page shows 'Rendering…' for a few seconds, then displays a lock icon and download button. To share publicly: tap the Share icon on a frozen clip and copy the /share/[token] link. Anyone with the link can listen without signing in. To revoke public access, tap Share again and toggle it off.

_Tags: ffmpeg, server-side, freeze, render, next.js-api, public-share_

### 🟡 Public Share Player — Unauthenticated Clip Playback

`odio-2026-015` · **active** · P3 · `components/share/PublicPlayer.tsx`

**Problem:** A frozen clip should be shareable outside the band — with a producer, a friend, or social media. The listener should not need a Google account or any signup. Only the frozen render is exposed; no session metadata, band data, or version history leaks through the public page.

**Analysis:**

The public share page (/share/[token]) fetches clip metadata by publicToken from Postgres. Audio is served via /api/audio/public/[token] — a separate proxy route that looks up the clip by token and streams the finalDriveFileId from Drive using the recorder's stored token. No user auth is required on either the page or the audio route.

PublicPlayer.tsx uses WaveSurfer.js (interact: false) to render the waveform and play the frozen audio. Stamp chips are rendered via StampJumpRow — tappable buttons that seek to their timestamp on tap, consistent with WaveformPlayer and WaveformEditor. WaveformPlayButton renders the shared amber play/pause button. A download button triggers the /api/audio/public/[token] route with a download intent, giving the user the .aac file.

The public page shows: clip name, band name (optional), clip duration, waveform player with stamps, play/pause, download. No version list, no vote buttons, no comments, no edit actions — nothing that reveals internal band structure.

**Implementation:**

1. components/share/PublicPlayer.tsx: client component. Props: token, sourceDurationMs, stamps[]. Uses useWaveSurfer({ url: /api/audio/public/{token}, interact: false — no cutMarksRef, no patchDurationUrl, public read-only }). Uses WaveformPlayButton for play/pause. Uses StampJumpRow for stamp chips (tappable, seek on tap) with custom className for layout. No WaveformCanvas overlay — no editing or seeking via waveform drag needed (play-only).
2. app/share/[token]/page.tsx: server component. Fetches clip by publicToken from Postgres (clips table). If not found or clip.frozen=false: returns 404 page. Renders clip name, duration, PublicPlayer with stamps.
3. app/api/audio/public/[token]/route.ts: streams audio from Drive. Looks up clip by publicToken, gets finalDriveFileId, fetches recorder's token from NextAuth accounts table, proxies Drive stream with Range header support. No auth middleware — open to public traffic.

**Key Decisions:**

- **Audio served via server-side proxy even on public pages**
  > Even with Drive public permission enabled on the final file, the Drive URL format includes the file ID which reveals internal structure. Proxying via /api/audio/public/[token] keeps the only public identifier as the opaque token, allows revocation (clear token → proxy returns 404), and avoids exposing file IDs in the page source.
- **No version history, no collaboration data on the public page**
  > The public page is for external listeners — producers, fans, collaborators outside the band. Exposing version numbers, vote tallies, or comments reveals internal band workflow details. The public page is intentionally minimal: name, waveform, stamps, download.

**Known Limitations:**
- Public audio proxy uses the recorder's Drive token — if the recorder revokes access, public links break even for frozen clips
- No view analytics — there's no way to know how many people have listened to a shared clip
- PublicPlayer renders the waveform but the overlay div is not present — users cannot seek by clicking on the waveform (play/pause only). Seeking via stamps is available.

**User Guide:** After freezing a clip, tap 'Enable public share' on the clip detail page to generate a /share/[token] link. Share this URL with anyone — they can open it in any browser without signing in. They'll see the clip name, a waveform player, and stamp markers. They can play, seek, and download the audio. To revoke access, tap 'Enable public share' again to disable — the link immediately stops working.

_Tags: public-share, wavesurfer, unauthenticated, frozen, stamps, download_

### 🔒 Storage Management — Drive Quota, Cleanup, Archive

`odio-2026-017` · **frozen** · P3 · `app/api/bands/[bandId]/storage/route.ts`

**Problem:** Audio files in Google Drive grow over time (~43MB per raw clip, ~10MB per frozen render). The recorder's free Drive tier (15GB) supports roughly 350 raw clips before action is needed. The app must surface this, and provide a clean way to free space after songs are finalized.

**Analysis:**

Storage indicator shown in band settings and session list: total Drive storage used by Odio files for this band (queried via Drive API quotas endpoint). Post-freeze cleanup: once a clip is frozen and the band is satisfied, the source .aac (raw recording) can be deleted from Drive — the frozen render and Postgres metadata remain. Archive: delete all versions of a clip from Postgres + Drive (both source and render). Bulk cleanup: session-level 'Archive all frozen clips' removes source audio for all frozen clips in a session at once.

**Implementation:**

1. app/api/bands/[bandId]/storage/route.ts: GET — fetches Drive API about endpoint using creator's token (returns storageQuota.usage). Also queries Postgres for all driveFileId + finalDriveFileId for this band's clips. Returns { usedBytes, limitBytes, fileCount }. Cached in memory for 5 minutes.
2. app/api/clips/[clipId]/cleanup/route.ts: POST — deletes clip.driveFileId from Drive (source .aac). Sets clip.driveFileId=null. Requires clip.frozen=true.
3. app/api/clips/[clipId]/route.ts DELETE: RECORDER only. Deletes clip from Postgres (cascades to versions, votes, comments). Deletes both driveFileId and finalDriveFileId from Drive via lib/drive.ts deleteFile.
4. components/bands/BandSettingsClient.tsx: fetches /api/bands/[bandId]/storage on mount. Renders storage bar (used / total). Lists members with role management. formatBytes helper: converts bytes to human-readable MB/GB string.
5. components/clips/ClipActionsClient.tsx: shows 'Clean up source audio' button when clip.frozen=true and clip.driveFileId is not null. Shows 'Delete clip' button (RECORDER only). Confirmation bottom sheets for both.

**Key Decisions:**

- **Cleanup is explicit and manual — no automatic deletion**
  > Automatic deletion after freeze could destroy recordings the band wanted to keep. Storage cleanup is a deliberate action. The app nudges (shows quota, shows how much can be freed) but never deletes automatically.

**Known Limitations:**
- Drive API quota query may not reflect real-time usage — cache with 5-minute TTL
- After cleanup, v1 source audio is gone — if the frozen render was also deleted and the band wants to re-edit, they would need to re-record
- Drive storage is the recorder's personal quota — shared with all their other Drive content

**User Guide:** In the Band tab, tap Settings to see Drive storage usage. The bar shows how many MB/GB of Drive are used by this band's recordings. After a clip is frozen and the band is happy with it, tap 'Clean up source audio' on the clip detail page to delete the raw recording from Drive — the final rendered file is kept and the clip remains fully playable. Tap 'Delete clip' (Recorder only) to permanently remove a clip and all its files from Drive.

_Tags: storage, google-drive, cleanup, archive, quota_
